# 产品需求文档 - infitek_erp

**作者：** friday
**日期：** 2026-04-13

## 执行摘要

infitek_erp 是星辰科技自主研发的 ERP 系统，以**单一数据来源（Single Source of Truth）**取代当前碎片化的 Excel + 宜搭低代码流转，覆盖销售、商务、采购、仓储全链路。系统面向 30+ 人业务团队，出口覆盖 160+ 国家，SKU 超 11,000 个，18 家国内供应商按单采购。

**MVP 目标**：建立 15 类核心主数据，并以简版交易模块打通完整业务链路：销售订单 -> 发货需求 -> 采购订单 -> 收货入库（补货路径）；发货需求 -> 物流单 -> 发货出库（发货路径）。发货需求作为核心枢纽，串联采购补货和物流发货两条路径。

**目标用户**：销售员（录单、查询）；采购员/助理（供应商管理、采购单）；商务跟单（发货需求、物流单协调）；仓库人员（收发货、库存）；产品专员（SKU 维护）；管理层（数据消费者）。

**系统全局能力版图**（MVP 标注 + Post-MVP 规划）：

| 能力域 | MVP | Post-MVP |
|--------|-----|----------|
| 主数据 | 15 类核心主数据 CRUD（产品分类/SPU/SKU/FAQ/证书库/资料库、供应商、客户、公司主体、港口、物流供应商、合同条款、单位/仓库/币种/国家） | 批量导入导出 |
| 销售 | 销售订单（简版，无审批流；订单类型字段含普通/售后/样品，MVP 仅启用普通订单） | 售后订单、样品订单类型开放、销售价格管理（含维护状态）、审批流 |
| 商务 | 发货需求（枢纽）、物流单（简版） | 物流费用、报关/清关单据打印、船期跟踪 |
| 采购 | 采购订单（简版，无审批流） | 采购申请单、PO合同审批、采购价格管理 |
| 库存 | 收货入库（简版）、发货出库（简版）、库存台账（SKU+仓库+入库批次维度，含锁定量；FIFO 自动分配） | 质检整改、完整批次管理（手动命名批次/批次报表/跨批次调拨）、收货/发货通知单 |
| 财务 | -- | 收款认领、资金账户、应付管理 |
| 系统 | 搜索与筛选 | RBAC 权限、CRM 集成、BI 看板 |

### 核心差异化

**单一数据来源，消除冗余副本**：SKU 报关信息、供应商账期、合同条款只在一处维护，下游所有单据通过关联引用获取。当前宜搭 ERP 中发货需求单高达 138 个字段，大量是从销售订单手工复制的冗余，这是数据不一致和报关差错的根源。

**发货需求作为业务枢纽**：发货需求单串联销售订单、采购订单、物流单和出库单，是整个业务链路的决策中心 -- 在这里判断"走库存"还是"走采购"，在这里触发物流安排和仓库出库。

**匹配真实业务流程**：数据模型精确对应星辰科技的出海贸易场景（HS 码、多主体签约、多币种、按单采购），不为通用 ERP 的无关功能付出复杂度代价。

## 项目分类

- **项目类型**：SaaS B2B（单租户内部系统）
- **领域**：企业运营管理（出口贸易进销存）
- **复杂度**：高（多模块交叉、出口合规、多主体运营）
- **项目上下文**：Brownfield（基于已有宜搭 ERP 需求参考和 Phase 1 PRD，全新自研系统）

## 成功标准

### 用户成功

- 销售员和采购员可在系统内直接查到完整 SKU 参数（规格、HS 码、重量体积），无需询问同事或翻 Excel
- 采购员可在供应商详情页一次性获取账期、联系方式、合同条款，无需翻 Excel
- 从销售订单到发货出库的完整业务流程可在系统内操作，不再通过微信或 Excel 传递信息
- 商务跟单可通过发货需求一目了然看到每个 SKU 的库存状态和采购进度，快速决策"走库存"还是"走采购"
- 仓管人员可通过物流单明确知道要为哪批货物出库，不再依赖微信确认
- 产品专员可在系统内统一管理产品证书和资料，证书有效期一目了然

### 业务成功

**核心指标一：主数据全面线上化** -- 15 类核心主数据全部在系统内维护，主数据的新增、修改、查询操作不再依赖 Excel。

- 产品分类、SPU、SKU（含 HS 码/报关信息）在系统内完整维护
- 产品证书库（CE/FCC 等）和产品资料库（说明书/规格书）在系统内管理
- 18 家供应商档案（含账期）全部入库
- 港口、物流供应商、公司主体信息完整录入
- 客户主数据建立

**核心指标二：主流程端到端可跑通** -- 销售订单 -> 发货需求 -> 采购订单 -> 收货入库 -> 物流单 -> 发货出库完整链路可在系统内操作，库存数量随收发货实时更新。

### 技术成功

- 所有 15 类主数据实体支持完整的 CRUD 操作
- 主数据字段模型支持交易模块引用，无需在单据中重复录入
- 库存台账支持期初库存录入，收发货操作实时触发库存变动，账面误差为 0
- 发货需求作为枢纽，正确串联上下游单据（销售订单、采购订单、物流单、出库单）

|------|------|
| 主数据实体类型 | 15 类全部上线 |
| 供应商档案入库 | 18 家（100%） |

| 主流程可跑通 | 销售订单->发货需求->采购订单->收货入库->物流单->发货出库全链路可操作 |

## 产品范围

### MVP -- 最小可行产品

**主数据（15 类）：**
- 产品体系：产品分类树（三级）、SPU、SKU（含 HS 码/报关字段）、FAQ、产品证书库、产品资料库
- 供应链参考：供应商档案（含账期/合同条款关联）、合同条款范本、港口信息、物流供应商
- 客户与公司：客户主数据、公司主体信息（含合规字段）
- 基础参考数据：单位、仓库、币种、国家/地区

**交易模块（简版）：**
- 销售订单：创建即确认（填写表单提交直接生成已确认记录）/查看，关联客户和 SKU 行项，创建后自动生成发货需求，无审批流
- 发货需求：由销售订单自动生成；展示 SKU 应发量与当前库存；作为创建采购订单（可多张，按供应商分组）和物流单的枢纽入口
- 采购订单：创建即确认（填写表单提交直接生成已确认记录）/查看，可从发货需求预填，关联供应商，无审批流
- 物流单：从发货需求创建，记录物流方式/供应商/装箱信息，作为出库依据
- 收货入库：关联采购订单确认收货，触发库存增加
- 发货出库：从物流单创建，确认发货触发库存扣减，校验库存充足

**基础能力：**
- 每个模块内关键字搜索 + 多条件筛选
- 库存台账：期初库存录入，SKU+仓库维度，收发货实时更新

### Growth Features（Post-MVP 近期）

- 批量导入/导出（所有主数据 Excel 模板导入、数据导出）
- RBAC 权限管理（自定义角色、模块级权限分配）
- 采购申请单、PO 合同审批流程
- 销售价格管理（价格记录含"待维护/已维护"状态，支持按状态筛选和批量维护）、采购价格管理
- 物流费用管理
- 变更审计日志

- 收货通知单 / 发货通知单（仓库操作的前置预通知层，MVP 跳过，由确认的采购订单/物流单直接驱动仓库操作）
- 销售订单售后类型、样品类型开放（MVP 仅支持普通订单类型）

### Vision（远期愿景）

- 销售订单/采购订单审批流
- 质检整改流程、完整批次管理（手动命名批次、批次报表、跨批次调拨）

### 旅程 4：Tom + Lily + 小王 + Sam -- "从下单到发货，第一次全程跑通"

**背景**：主数据已录入完毕，业务团队第一次在 ERP 中完整走一笔业务：客户 ABC Trade 下了 20 套 LED 洗墙灯订单。

**旅程**：

**Step 1 -- 销售员 Tom 创建销售订单**
- 新建销售订单，关联客户（ABC Trade），添加订单行：LED 洗墙灯 50W x 20，填写单价
- 确认提交，状态变为"已确认"
- 系统自动生成对应的**发货需求单**，包含：LED 洗墙灯 x 20

**Step 2 -- 发货需求单成为操作枢纽**
- 商务跟单小王打开发货需求单，查看每个 SKU 的应发数量和当前库存
- 系统显示：LED 洗墙灯当前库存 0，库存不足

**Step 3 -- 采购员 Lily 完成采购订单**
- 采购订单已从发货需求单预填 SKU 和数量，Lily 补充供应商（信达包装）和单价
- 确认提交，状态变为"已确认"

**Step 4 -- 仓管 Sam 收货入库**

**Step 5 -- 商务跟单小王安排物流**
- 回到发货需求单，此时库存已满足
- 小王从发货需求单操作"创建物流单"
- 填写物流供应商、运输方式、装箱信息（箱数、尺寸、毛重）
- 物流单创建完成，作为仓库出库的依据

**Step 6 -- 仓管 Sam 从物流单发货出库**
- Sam 打开物流单，操作"创建发货出库单"
- 填写实发数量 20，来源仓库（深圳仓），确认发货
- 系统库存 -20；销售订单状态更新为"已发货"

**结果**：完整业务链路在系统内留存记录，发货需求单串联了销售订单、采购订单、物流单和出库单，库存账面与实物一致，全程无 Excel 和微信。

**揭示的需求**：销售订单简版 CRUD、发货需求（枢纽）、采购订单简版 CRUD、物流单简版 CRUD、库存台账、收货入库单、发货出库单、库存实时更新

## 领域特定需求

### 出口合规字段

- **HS 码**：SKU 上的必填字段（8 或 10 位数字）；字段格式校验（纯数字位数检查）列为 Post-MVP
- **报关信息字段**：SKU 需维护报关品名（中英文）、申报价值参考，作为出口报关数据来源
- **分类准确性**：HS 码的分类准确性由录入人员负责，MVP 不接入外部海关数据库
- **产品证书**：CE/FCC 等证书在产品证书库中维护，含有效期、发证机构、适用 SPU；证书到期自动预警列为 Post-MVP

### 公司主体合规字段

公司主体信息需完整录入：纳税人识别号、海关备案号、检疫备案号、开户行/银行账号/SWIFT CODE、默认币种、签订地点。多公司主体运营场景下，采购订单和物流单需关联正确的签约主体。

### 多币种支持

- 销售订单支持外销币种（USD、EUR 等），系统维护币种基础数据
- MVP 阶段不做汇率自动换算，订单金额以录入币种为准
- Post-MVP：汇率维护表 + 财务多币种对账

### 租户模型

infitek_erp 是星辰科技内部**单租户系统**，不对外提供 SaaS 服务：单一组织、单一数据空间、单一部署实例，无多租户隔离需求，无订阅计划或计费模块。

### 权限模型

- **MVP**：不实现 RBAC，所有登录用户有全系统操作权限
- **Post-MVP**：自定义角色 + 模块级权限（读/写/删除），超级管理员角色内置

### 集成清单

- **MVP**：无外部系统集成，系统独立运行
- **Post-MVP**：与 CRM（富通天下）定时同步，拉取销售订单至 ERP，消除"导出-补充"断点

### 技术架构方向

Web 浏览器优先（桌面），支持 Chrome / Edge；移动端不在 MVP 范围；技术栈选型由架构师决策。

## 项目类型专项需求

### 技术架构考量

**数据模型复杂度**：15 类主数据实体 + 6 类交易单据 + 库存台账，实体间存在大量关联引用（SKU -> SPU -> 分类，采购订单 -> 供应商，物流单 -> 发货需求 -> 销售订单）。数据模型设计需确保关联引用的一致性，避免重现宜搭的字段冗余问题。

**状态机管理**：交易单据（销售订单、采购订单、发货需求、物流单）均有明确的状态流转。MVP 阶段状态机保持简版（具体状态定义由架构设计阶段确定）：
- 销售订单：**已确认 -> 已发货**（草稿为前端表单填写中间状态，不持久化至数据库；用户填写完整后提交，系统直接创建为"已确认"状态记录）
- 采购订单：**已确认 -> 已收货**（草稿为前端表单填写中间状态，不持久化至数据库）
- 发货需求：**待分配库存**（初始状态）-> 待生成采购单（存在采购缺口但尚未生成采购订单）-> 采购中（已有有效请购型采购订单）-> 备货完成 -> 部分发货 -> 已发货 | 已作废；**作废仅在"待分配库存"初始状态下允许**，作废不可逆，自动释放锁定库存
- 物流单：**已确认 -> 已出库**（草稿为前端表单填写中间状态，不持久化至数据库；物流跟踪字段可在确认后于详情页补填）

**库存数量模型**：库存以双层结构维护，数据模型须支持原子性更新：
- **汇总层（SKU + 仓库）**：聚合三个数量字段：**实际库存**（所有批次入库之和减去已出库总量）、**锁定量**（被发货需求预占但尚未出库的数量）、**可用库存**（= 实际库存 - 锁定量）
- **批次层（SKU + 仓库 + 入库批次）**：每次收货入库单确认或期初库存录入时创建一条批次记录，含入库日期、来源单据（采购订单号或"期初录入"）、该批次实际数量、该批次锁定量；锁定明细记录到具体批次
- **FIFO 分配**：系统执行库存锁定时，按入库日期从早到晚自动跨批次分配锁定量，无需用户手动选批次；发货出库确认时同样按 FIFO 顺序扣减批次实际库存

**库存锁定触发时机**：
- 走库存路径：用户在发货需求 SKU 行选择"使用现有库存"或"部分采购"（库存部分），填写使用数量确认后，系统立即按 FIFO 跨批次锁定对应数量
- 走采购路径（请购型）：采购订单收货确认时，系统按 FIFO 自动将入库数量锁定至来源发货需求
- 锁定释放：发货出库确认后（汇总层 + 批次层实际库存 -，锁定量 -）；发货需求作废时（锁定量 - 仅，对应批次锁定明细清除）

**单据关联链路**：销售订单（1）-> 发货需求（1:1）-> 采购订单（N）+ 物流单（N）-> 出库单（1:1 物流单）；收货入库单（N）-> 采购订单（1）。MVP 阶段简化为主路径关联，不处理拆单/合单场景。

### 实现考量

**组件复用策略**：15 类主数据模块的 UI 模式高度一致（列表页 + 详情页 + 表单页），应抽象标准化 CRUD 组件，降低开发量和维护成本。

**搜索与筛选**：每个模块需支持关键字搜索（至少匹配名称字段）和多条件组合筛选。列表页默认分页加载。

**审计基础**：MVP 不要求变更审计日志，但数据模型应预留 created_at、updated_at、created_by、updated_by 字段，为 Post-MVP 审计功能做准备。

## 项目范围界定与阶段规划

### MVP 策略

**MVP 哲学：问题解决型 MVP（Problem-Solving MVP）**

infitek_erp 的 MVP 不追求功能全面，而是聚焦一个核心命题：**让业务团队在系统内完成一笔完整的订单生命周期，不再依赖 Excel 和微信**。

MVP 的验证目标：
1. 主数据是否能取代 Excel 成为唯一数据来源？
2. 交易链路是否能端到端跑通？
3. 库存账面是否与实际操作一致？

如果这三个问题的答案都是"是"，MVP 就成功了。

### MVP 支撑的核心旅程

| 旅程 | MVP 支撑程度 |
|------|------------|
| 旅程 1（Lily - 供应商录入） | 完整支撑 |
| 旅程 2（Jason - 产品档案建立） | 完整支撑 |
| 旅程 3（Admin - 系统初始化） | 完整支撑 |
| 旅程 4（端到端交易链路） | 完整支撑（简版，无审批） |

### 明确不在 MVP 的决策及理由

| 排除项 | 理由 | 替代方案 |
|--------|------|---------|
| 审批流（销售/采购） | 增加流程复杂度，MVP 阶段信任人工判断 | 线下沟通确认后在系统操作 |
| 财务模块 | 独立性高，不阻塞主流程 | 继续使用现有财务流程 |
| 批量导入/导出 | 非日常操作，初始化可手动录入 | 逐条录入或由运维脚本辅助 |
| CRM 集成 | 外部依赖，接口对接周期长 | 销售员在 ERP 手动录入订单 |
| 质检整改流程 | 收货简化为"确认即入库" | 线下质检通过后再在系统确认收货 |
| 收货通知单 / 发货通知单 | MVP 以"已确认采购订单"和"已确认物流单"直接驱动仓库操作，无需前置通知层 | Post-MVP 补充通知单层 |
| 完整批次管理 | MVP 已引入轻量批次层（收货即批次 + FIFO）满足发货需求；手动命名批次、批次报表、跨批次调拨等管理功能增加额外复杂度 | Post-MVP 补充完整批次管理功能 |

### 风险与应对

| 风险类型 | 具体风险 | 应对措施 |
|---------|---------|---------|
| 技术风险 | 单据关联链路复杂，状态流转出错 | MVP 简化状态机，主路径优先，边界场景后补 |
| 技术风险 | 物流单与出库单关联复杂度 | MVP 阶段简化为一对一关系，一个物流单对应一个出库单 |
| 采纳风险 | 用户习惯 Excel，抗拒切换 | 主数据先行，让用户在查询场景感受到系统价值 |
| 数据风险 | 期初数据录入错误 | 提供汇总核对视图，业务方确认后才投入使用 |
| 数据风险 | 无 RBAC 阶段误操作主数据 | 编辑操作需二次确认；Post-MVP 上线 RBAC 后收紧权限 |
| 范围风险 | 15 类主数据 + 交易模块工作量大 | 标准化 CRUD 组件复用；主数据和交易模块可分批上线 |

### 6. 销售订单管理

- **FR23**：授权用户可查看销售订单列表（支持按状态、客户筛选）和订单详情
- **FR24**：销售订单确认后，授权用户可在销售订单详情页点击"生成发货需求"按钮，手动触发生成对应的发货需求单；发货需求包含订单中所有 SKU 及应发数量；发货需求不支持独立手动创建，唯一来源为从已确认销售订单手动触发；生成时系统自动查询每个 SKU 在各仓库的当前可用库存（= 实际库存 - 锁定量），随应发数量一并在发货需求详情中展示
- **FR25**：系统在关联发货出库单全部确认发货后，自动将销售订单状态更新为"已发货"

### 7. 发货需求与物流（业务枢纽）

- **FR26**：发货需求详情展示每个 SKU 行的应发数量及各仓库当前可用库存（= 实际库存 - 锁定量）；每个 SKU 行含**履行类型**字段（枚举，用户在详情页逐行设置），三个值：`全部采购`（全部数量走采购，纳入"创建采购订单"范围）、`部分采购`（用户分别填写使用库存数量和采购数量，库存部分立即锁定，采购部分纳入"创建采购订单"范围）、`使用现有库存`（全部数量使用库存，不纳入采购）；选择"使用现有库存"或"部分采购"时，SKU 行展开内联仓库分配区，展示该仓库下各入库批次的明细（入库日期、来源单据、可用数量），用户填写使用数量并确认后，系统按 **FIFO**（入库日期从早到晚）自动跨批次完成**库存锁定**，用户无需手动指定批次
- **FR27**：授权用户可从状态为"待生成采购单"或"采购中"的发货需求单创建采购订单；采购订单行仅包含履行类型为"全部采购"或"部分采购"且仍有可下单数量的 SKU，可下单数量 = `purchaseRequiredQuantity - purchaseOrderedQuantity`，不可超过采购缺口；此类采购订单为**请购型**，创建成功后聚合回填 `purchaseOrderedQuantity`，并将"待生成采购单"的发货需求推进为"采购中"；收货后自动触发库存锁定（详见 FR45）；**一个发货需求可创建多张采购订单**（通常按供应商分组，每张采购订单关联一个供应商），已有关联采购订单不影响再次从同一发货需求创建新的采购订单，以满足向不同供应商分别下单的业务场景
- **FR28**：授权用户可从发货需求单创建物流单；物流单只允许使用已锁定且尚未被 active 物流单计划的待发数量，不得使用待采购数量或尚未收货锁定的采购缺口
- **FR29**：授权用户可查看发货需求单列表（支持按状态、关联销售订单筛选）和详情
- **FR30**：发货需求单状态随关联出库确认自动更新：部分 SKU 出库后状态变为"部分发货"，全部 SKU 出库完成后状态变为"已发货"
- **FR31**：授权用户可创建物流单，关联发货需求单；字段分组如下：
  - **基本信息**（必填）：物流供应商、运输方式（海运/陆运/空运/快递）、公司主体（签约主体）、起运港（关联港口主数据）、目的港/目的地（关联港口主数据）、运抵国
  - **物流跟踪**（确认后补填）：预计离港日期（ETD）、预计到港日期（ETA）、订舱号、船司/航司、船名航次、SO/提单号、实际离港日期
  - **装箱信息**（子行项，必填）：SKU、箱数、每箱数量、长×宽×高（cm）、毛重（kg）
  - **报关信息**（选填）：是否出口报关、唛头
  - **备注**：其他要求及注意事项
- **FR32**：授权用户通过提交物流单表单完成创建（**草稿不入库**），系统直接生成"已确认"状态的物流单记录；物流跟踪字段（ETD/ETA/订舱号/船名航次/SO 号等）在确认后可在物流单详情页随时补填或更新
- **FR33**：授权用户可从物流单创建发货出库单
- **FR34**：授权用户可查看物流单列表（支持按状态筛选）和详情

### 8. 采购订单管理

- **FR35**：授权用户可通过表单创建采购订单（**草稿不入库**，表单填写过程不产生数据库记录），填写完整后提交，系统直接创建为"已确认"状态；可从发货需求单预填 SKU 和数量（请购型），默认数量取 `purchaseRequiredQuantity - purchaseOrderedQuantity`；创建成功后回填 `purchaseOrderedQuantity` 并把发货需求从"待生成采购单"推进为"采购中"；关联已有供应商，补充单价；合同条款字段默认关联该供应商下状态为"已审批"的第一条合同条款范本（见 FR09），用户可手动修改；采购订单须在创建时通过类型字段标注为"请购型"或"备货型"，影响收货后的库存锁定行为（见 FR45）
- **FR36**：采购订单通过表单提交直接创建为"已确认"状态，无独立的从草稿变更为已确认的操作步骤；采购订单一经创建即为已确认，不支持撤销至草稿
- **FR37**：授权用户可查看采购订单列表（支持按状态、供应商筛选）和订单详情
- **FR38**：系统在关联收货入库单确认收货后，自动将采购订单状态更新为"已收货"

### 9. 库存管理

- **FR39**：系统以双层结构维护库存：**汇总层**（SKU + 仓库）含三个聚合字段：实际库存、锁定量、可用库存（= 实际库存 - 锁定量）；**批次层**（SKU + 仓库 + 入库批次）每条记录含入库日期、来源单据（采购订单号或"期初录入"）、该批次实际数量、该批次锁定量；两层数据须保持一致（汇总层 = 批次层聚合）
- **FR40**：系统管理员可为指定 SKU 和仓库录入或覆盖期初库存数量，用于系统上线时初始化账面库存；期初录入确认后在批次层创建一条来源标注为"期初录入"的批次记录；期初录入支持重复提交以覆盖已有数据（原期初批次记录随之更新）；系统不提供日常库存调整（盘点差异）功能，库存数量仅通过收货入库和发货出库操作变动
- **FR41**：系统在发货出库操作确认后，实时扣减对应 SKU 在对应仓库的实际库存，并释放对应锁定量
- **FR42**：系统自动记录每次库存变动流水（变动类型、SKU、仓库、数量变化、操作人、操作时间）；变动类型包括：期初录入、采购入库、发货出库、锁定、解锁
- **FR43**：授权用户可按 SKU 或仓库查询当前库存数量（实际/锁定/可用）及历史变动流水
- **FR44**：授权用户可创建收货入库单，关联已确认的采购订单，逐行填写每个 SKU 的实际收货数量和目标仓库；收货确认后在批次层创建对应批次记录（来源标注为该采购订单号，入库日期为确认操作日期）
- **FR45**：授权用户确认收货后，系统自动触发：① 在批次层创建新批次记录并同步更新汇总层实际库存；② 根据采购订单类型分别处理锁定：**请购型**（由发货需求驱动创建，见 FR27）收货后系统按 **FIFO**（入库日期从早到晚）自动跨批次将入库数量**锁定至对应发货需求**，锁定明细写入对应批次记录；**备货型**（主动备货采购，非发货需求驱动）收货后仅增加实际库存，不触发锁定；③ 采购订单状态更新为"已收货"
- **FR46**：授权用户可从物流单创建发货出库单，逐行填写每个 SKU 的**实际发货数量**（允许部分发货，实际数量可少于物流单计划数量）和来源仓库；库存扣减以实际发货数量为准
- **FR47**：系统在创建发货出库单时校验出库数量不超过该仓库中该 SKU **被本发货需求锁定的数量**；出库数量超出本需求锁定量时阻止提交，并提示当前本需求锁定数量；逻辑依据：库存在 FR26 时已按 FIFO 为本需求预留，出库只消耗本需求已预留的库存，不额外占用其他发货需求的锁定量或自由可用量
- **FR48**：授权用户确认发货后，系统自动触发对应 SKU 在来源仓库的实际库存扣减，并释放对应的锁定量
- **FR49**：授权用户可对**状态为"待分配库存"**（初始状态，尚未确认分配）的发货需求执行**作废**操作；处于"待生成采购单"、"采购中"、"备货完成"、"部分发货"、"已发货"状态的发货需求**不允许作废**，系统应提示原因并阻止操作；作废需二次确认（提示"作废后将释放所有锁定库存，操作不可撤销"）；作废确认后，系统自动触发：① 解锁该发货需求占用的所有锁定库存；② 关联销售订单状态回退为"已确认"；③ 系统在关联销售订单详情页展示"重新生成发货需求"按钮，用户可手动触发重新生成新的发货需求（重走 FR24 逻辑）；作废后该发货需求状态变为"已作废"，不可再操作

## 非功能需求

### 性能

- **NFR-P1**：主数据列表页首屏加载时间 < 2 秒（标准网络，100 条以内数据）
- **NFR-P2**：主数据详情页加载时间 < 1.5 秒
- **NFR-P3**：搜索和筛选操作响应时间 < 1 秒
- **NFR-P4**：销售订单、采购订单提交（状态变更）响应时间 < 2 秒
- **NFR-P5**：收货入库、发货出库确认操作（含库存更新）响应时间 < 3 秒
- **NFR-P6**：库存查询（当前库存数量）响应时间 < 1 秒

### 安全

- **NFR-S1**：所有数据传输使用 HTTPS 加密
- **NFR-S2**：用户密码以不可逆加密方式存储，不存储明文
- **NFR-S3**：用户连续无操作超过 8 小时后自动登出（时长可配置）
- **NFR-S4**：所有接口须通过身份验证，系统拒绝未授权请求

### 可靠性

- **NFR-R1**：系统目标可用性 99%（允许计划内维护窗口）
- **NFR-R2**：数据每日自动备份，支持按需恢复

### 可用性体验

- **NFR-U1**：系统面向桌面浏览器优先，支持 Chrome / Edge 最新两个主版本
- **NFR-U2**：所有用户操作的错误提示须清晰说明出错原因，不暴露技术错误码或堆栈信息
- **NFR-U3**：发货出库库存不足的校验提示须明确显示当前可用库存数量，方便用户直接判断

### 文件存储

- **NFR-F1**：产品资料库和产品证书库的文件上传及读取使用阿里云 OSS；系统服务端不在本地磁盘持久化文件内容
- **NFR-F2**：文件删除执行物理删除（从 OSS 永久删除），不保留软删除记录；删除操作需二次确认
