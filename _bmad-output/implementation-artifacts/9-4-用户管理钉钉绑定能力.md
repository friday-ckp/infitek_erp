# Story 9.4: 用户管理钉钉绑定能力

Status: done

## Story

As a 系统管理员,
I want 在用户管理中查看并维护钉钉绑定关系,
so that 我可以控制哪些企业员工可以通过钉钉扫码登录系统。

## Acceptance Criteria

1. **列表与详情展示绑定状态**
   - 管理员进入用户管理列表或详情时，可看到每个用户的钉钉绑定状态
   - 状态至少区分“未绑定”和“已绑定”
   - 已绑定用户可看到绑定时间

2. **展示最小必要识别信息**
   - 查看已绑定用户时，可看到用于识别的钉钉展示信息，至少包含昵称
   - 不暴露 `dingtalk_union_id`、`dingtalk_user_id`、`dingtalk_open_id`、access token 或其他敏感凭据
   - 头像是可选展示项，不是必需交付项

3. **管理员可为本地用户建立绑定**
   - 系统提供绑定接口和前端维护入口
   - 绑定时保存 `dingtalk_union_id`，并可同时保存 `dingtalk_user_id`、`dingtalk_open_id`、`dingtalk_nick`、`dingtalk_avatar`
   - 绑定成功后写入 `dingtalk_bound_at`
   - 绑定成功后该用户可复用 Story 9.2 已完成的扫码登录链路

4. **同一钉钉身份不可重复占用**
   - 若某个 `dingtalk_union_id` 已绑定到其他用户，系统拒绝再次绑定
   - 返回稳定业务错误码和统一错误响应
   - 前端展示明确中文提示“该钉钉账号已绑定其他 ERP 用户”

5. **管理员可解绑**
   - 管理员确认解绑后，系统清空该用户全部钉钉绑定字段
   - 解绑后该钉钉身份不能再直接扫码登录
   - 页面给出成功反馈

6. **保持现有权限边界并落实临时管理员规则**
   - 绑定/解绑接口必须要求已登录用户访问
   - 在当前无 RBAC 的仓库现实下，本 Story 以 `currentUser.username === 'admin'` 作为临时管理员判定
   - 非 `admin` 用户访问绑定/解绑接口时返回 `403`
   - 前端对非 `admin` 用户隐藏或禁用绑定维护动作，但不需要在本 Story 中重做整套用户管理权限体系

## Dev Notes

### Story Boundary

- 本 Story 只补“用户管理中的钉钉绑定状态展示 + 管理员绑定/解绑能力”。
- 本 Story 不修改 Story 9.2 已完成的钉钉 OAuth 协议，不重写 `/auth/dingtalk/login`、`/auth/dingtalk/callback`、`/auth/dingtalk/exchange`。
- 本 Story 不新增自动开户、扫码即绑定、手机号批量预匹配 UI、独立 `user_identities` 表、RBAC 角色系统。
- Story 9.5 会对“绑定成功后可登录、解绑后不可登录”做联调验收；本 Story 先把绑定数据维护能力落地。

### Story Foundation And Cross-Story Context

- Epic 9 的核心原则是“钉钉只负责外部认证，本地系统继续签发 JWT”，且未绑定用户不得扫码直接进入系统。
- Story 9.1 已经把 `users` 表扩展为钉钉身份载体，并明确本阶段采用“管理员预绑定，不自动开户”。
- Story 9.2 已经把运行时匹配主键固定为 `dingtalk_union_id`，因此 9.4 绑定管理必须继续围绕这一字段，不允许另起身份模型。
- Story 9.3 已经把未绑定登录失败文案落到了前端回调页，因此 9.4 的绑定成功与解绑成功会直接改变 9.3 的用户体验结果。

### Critical Conflict Resolution

- 规划文档与早期 Story 1.4 多次使用“系统管理员”措辞，但当前代码库没有 RBAC、角色表或 `isAdmin` 字段。
- 当前 JWT payload 和 `JwtStrategy.validate()` 只回传 `{ id, username }`，因此本 Story 不能虚构一个尚不存在的角色系统。
- 为满足 AC 6 且不扩 scope，本 Story 的**临时管理员规则**必须明确写死为：`currentUser.username === 'admin'`。
- 这条规则只约束**新加的钉钉绑定/解绑接口与前端维护动作**；不要借本 Story 回头重构整个用户管理访问边界。
- 后续若项目真正上线 RBAC，应由独立 Story 替换这条临时规则。

### Current Code Context

- `apps/api/src/modules/users/entities/user.entity.ts` 已经持有全部钉钉字段：
  - `dingtalkUnionId`
  - `dingtalkUserId`
  - `dingtalkOpenId`
  - `dingtalkNick`
  - `dingtalkAvatar`
  - `dingtalkBoundAt`
- 但这些字段在当前 API 中尚未形成稳定的“绑定状态”响应契约，且敏感字段被 `@Exclude()`，不能指望前端直接拿到完整内部值。
- `apps/api/src/modules/users/users.controller.ts` 目前只有：
  - `GET /users`
  - `GET /users/:id`
  - `POST /users`
  - `PATCH /users/:id`
  - `POST /users/:id/deactivate`
  尚无钉钉绑定专用接口。
- `apps/api/src/modules/users/users.service.ts` 目前只有基础 CRUD 与 `findByDingtalkUnionId()` 查询；尚无 bind / unbind 业务方法。
- `apps/api/src/modules/auth/auth.service.ts` 已经在扫码登录回调中通过 `findByDingtalkUnionId()` 做登录匹配，因此 9.4 只需维护正确字段，不要再碰 Auth 主链路。
- `apps/web/src/pages/settings/users/index.tsx` 当前列表页使用本地 `useState + useEffect` 拉取数据；本 Story 不要顺手把整个用户列表重构成全新的数据流。
- `apps/web/src/pages/settings/users/detail.tsx` 已经有锚点式详情骨架，最适合新增“钉钉绑定”区块。
- `apps/web/src/api/users.api.ts` 当前 `User` 类型还没有钉钉绑定字段，需要补齐。

### Backend Guardrails

- 继续复用现有全局 JWT 守卫，不要把新接口做成 `@Public()`。
- 新接口路径必须遵循当前真实 API 前缀 `/api`，不要回退到历史文档里的 `/api/v1`。
- 不要通过修改 `User` 实体的 `@Expose()` 范围，把 `dingtalkUnionId` 等内部字段整体暴露给所有 `users` 响应。
- 若需要绑定状态，优先返回派生字段或最小 DTO，而不是把底层身份主键直接下发。
- 不要把钉钉绑定写进通用 `update()` 流程，避免未来用户编辑接口意外覆盖外部身份字段。
- 绑定/解绑都要记录 `updatedBy`，保持与现有审计字段一致。

### Frontend Guardrails

- 用户管理属于现有“基础数据 / 用户管理”页面体系，继续沿用当前列表页与详情页骨架，不要新建独立的“钉钉绑定中心”页面。
- 列表页只负责扫描状态；复杂维护动作收敛到详情页，避免行内堆叠按钮和弹窗。
- 当前 `useAuthStore()` 持久化的 `user` 只有 `username`、`name`，因此前端管理员判断也应基于 `user?.username === 'admin'`。
- 这只是前端体验层判断，真正权限控制必须仍在后端执行。
- 昵称足以满足 AC 2，头像不是必需项；如果展示头像，必须确保 URL 仅作为普通展示数据使用，不做任何鉴权含义。

### Recommended Implementation Shape

- 后端推荐新增 DTO：
  - `apps/api/src/modules/users/dto/bind-dingtalk.dto.ts`
- 后端推荐新增 service 方法：
  - `bindDingtalkIdentity(userId, dto, operatorUsername)`
  - `unbindDingtalkIdentity(userId, operatorUsername)`
- 后端推荐新增 controller 路由：
  - `POST /users/:id/dingtalk-binding`
  - `DELETE /users/:id/dingtalk-binding`
- 前端推荐新增 API 方法：
  - `bindUserDingtalk(id, payload)`
  - `unbindUserDingtalk(id)`
- 前端推荐在详情页新增“钉钉绑定”区块：
  - 状态 pill
  - 绑定时间
  - 昵称
  - admin 可见的“绑定钉钉账号”“解绑钉钉账号”按钮
- 绑定交互推荐采用简洁表单 Modal 或 Drawer，不要为了本 Story 引入额外子路由。

### Data Contract Guidance

- 列表与详情响应建议至少提供以下只读字段：
  - `dingtalkBindingStatus`
  - `dingtalkBoundAt`
  - `dingtalkNick`
- 绑定请求体建议：

```json
{
  "dingtalkUnionId": "ding-union-xxx",
  "dingtalkUserId": "optional-user-id",
  "dingtalkOpenId": "optional-open-id",
  "dingtalkNick": "张三",
  "dingtalkAvatar": "https://..."
}
```

- 错误码建议至少稳定以下两项：
  - `DINGTALK_IDENTITY_ALREADY_BOUND`
  - `DINGTALK_BINDING_FORBIDDEN`
- 对同一用户重复绑定同一 `dingtalkUnionId` 的情况，建议按“幂等更新辅助字段”处理，避免管理员二次保存时报错。

### Testing Guidance

- 后端重点验证：
  - 绑定写入后，`findByDingtalkUnionId()` 能查到目标用户
  - 解绑后，同一 `dingtalkUnionId` 不再命中该用户
  - 非 `admin` 用户无法调用绑定/解绑接口
- 前端重点验证：
  - `admin` 登录态下可见维护按钮，非 `admin` 不可见或不可用
  - 绑定占用错误能被正确映射
  - 解绑后页面状态从“已绑定”切回“未绑定”
- 联调层面的“解绑后扫码失败”完整链路由 Story 9.5 兜底，但本 Story 至少要保证绑定字段已被正确清空。

  - `feat(story-9.1): 钉钉应用配置与用户模型扩展`

### Technical Research Summary

- 钉钉接入方案已确定为“企业内部应用 + 网页应用能力 + 管理员预绑定 + 本地 JWT 签发”。
- 因此 9.4 的绑定动作本质上是**维护本地用户与外部身份的映射关系**，不是重新做一遍 OAuth 流程。
- 当前单租户阶段推荐直接在 `users` 表保存绑定信息，未来若接入多个外部 IdP 再演进成独立身份表。
