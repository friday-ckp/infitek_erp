# Story 9.1: 钉钉应用配置与用户模型扩展

Status: done

## Story

As a 开发者,
I want 先完成钉钉扫码登录接入所需的基础配置、用户模型扩展与迁移准备,
so that 后续 OAuth 回调、前端登录入口、绑定管理与联调测试可以建立在稳定一致的身份模型与环境约定之上。

## Acceptance Criteria

1. **接入模式定稿**
   - 明确采用“企业内部应用 + 网页应用 + 后端 OAuth 回调 + 本地 JWT 签发 + 管理员预绑定”的接入模式
   - 保留现有用户名密码登录链路作为兜底，不替换现有 JWT 认证体系

2. **环境变量与回调地址约定**
   - 后端定义 `DINGTALK_CLIENT_ID`、`DINGTALK_CLIENT_SECRET`、`DINGTALK_REDIRECT_URI`、`DINGTALK_FRONTEND_CALLBACK_URI`
   - 生产回调地址指向后端钉钉回调接口
   - 前端回调地址指向 `/login/dingtalk/callback`
   - 开发环境回调地址仅用于联调，不作为正式开放平台配置

3. **用户模型扩展**
   - `users` 表新增 `dingtalk_union_id`、`dingtalk_user_id`、`dingtalk_open_id`、`dingtalk_nick`、`dingtalk_avatar`、`dingtalk_bound_at`
   - `dingtalk_union_id` 作为运行时唯一外部身份主键
   - `dingtalk_union_id` 存在唯一索引

4. **绑定策略约定**
   - 默认采用“管理员预绑定，扫码只做认证，不自动开户”
   - 未绑定用户扫码登录时被拒绝，并提示联系管理员
   - 如需初始化导入，可允许一次性预匹配，但最终绑定仍需人工确认

5. **迁移与兼容性**
   - 存在对应的 TypeORM migration 方案以扩展 `users` 表
   - 不破坏现有本地登录、用户管理、JWT 校验与审计字段逻辑

6. **安全边界约定**
   - 明确后续登录成功后不直接把 JWT 放在 URL 上
   - 明确后续通过一次性 `loginTicket` 交换正式 JWT
   - 当前单租户阶段不额外拆分独立身份绑定表，后续多 IdP 再评估独立 `user_identities`

## Dev Notes

### Story Boundary

- 本 Story 不实现真实钉钉 OAuth 登录接口，也不实现前端回调页，只负责把“能不能接、字段怎么落、配置怎么约定、绑定策略是什么”这些基础条件打牢。
- 本 Story 也不在此阶段实现用户管理中的绑定/解绑交互，但字段模型与配置约定必须让 Story 9.4 可以直接承接。
- 如果需要在本 Story 中增加最小的配置模块或 migration 辅助代码，可以做；但不要提前实现完整 `AuthController` 钉钉登录入口。

### Current Code Context

- `apps/api/src/modules/users/entities/user.entity.ts` 当前已有 `username`、`name`、`password`、`status`、`deletedAt`，并继承 `BaseEntity` 审计字段，适合作为钉钉身份字段的承载实体。
- `apps/api/src/modules/auth/auth.controller.ts` 当前仅暴露 `POST /auth/login`，Story 9.1 不能破坏现有本地登录入口。
- `apps/api/src/modules/auth/auth.service.ts` 当前只包含 `login(username, password)`，并沿用系统 JWT。后续 Story 9.2 应在此基础上新增钉钉登录链路，而不是重写现有密码登录。
- `apps/web/src/pages/login/index.tsx` 当前仍保留“通过企业 SSO 登录”的占位按钮，后续 Story 9.3 会接入真实跳转。
- `apps/web/src/store/auth.store.ts` 当前只保存 `token` 和本地用户信息，后续钉钉登录成功后也应复用这一路径写入 JWT，而不是另建一套登录态。

### Architecture Constraints

- 认证策略继续使用系统内统一 JWT，密码登录和钉钉扫码登录最终都签发同一种系统 token。[Source: `_bmad-output/planning-artifacts/architecture.md#认证与安全`]
- 钉钉仅作为外部身份提供方，系统继续维护本地用户与权限语义，不把钉钉 token 直接当系统 token 使用。[Source: `_bmad-output/planning-artifacts/architecture.md#认证与安全`]
- 后续回调成功后不得把 JWT 直接挂在 URL 上，而是使用一次性 `loginTicket` 交换正式 JWT。[Source: `_bmad-output/planning-artifacts/architecture.md#认证与安全`]
