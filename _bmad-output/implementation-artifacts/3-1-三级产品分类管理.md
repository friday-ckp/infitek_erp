# Story 3.1: 三级产品分类管理

Status: ready-for-dev

## Story

As a 产品专员,
I want 创建和维护最多三级的产品分类树（如"灯具 > 户外灯 > 洗墙灯"）,
So that 产品档案有清晰的层级分类结构，后续 SPU 可以归属到正确的分类下。

---

## Acceptance Criteria

**Given** 产品专员进入产品分类管理页面
**When** 页面加载完成
**Then** 展示产品分类树形结构（Tree 左侧面板 + 右侧列表双面板布局）
**And** 支持展开/折叠各级分类节点

**Given** 产品专员点击"新建分类"
**When** 填写分类名称，选择父级分类（可选，不选则为一级分类）
**Then** 分类创建成功，树形结构实时刷新
**And** 最多允许三级层级，第三级分类下不允许再创建子分类（"新建子分类"按钮禁用）

**Given** 产品专员点击某个分类节点
**When** 查看分类详情（右侧面板）
**Then** 展示分类名称、层级路径（如"灯具 / 户外灯"）、下属 SPU 数量（当前阶段显示 0）
**And** 可点击"编辑"修改分类名称

**Given** 产品专员编辑分类名称
**When** 保存修改
**Then** 树形结构中对应节点名称同步更新

**Given** 后端分类 API
**When** 查看接口设计
**Then** 分类实体包含 id、name、parentId、level、sortOrder、createdAt、updatedAt、createdBy、updatedBy 字段
**And** 提供 `GET /product-categories/tree` 返回完整嵌套树（一次加载全部数据）
**And** 提供标准 CRUD 接口

## Dev Agent Context

### 核心差异：这不是标准 CRUD 模块

Story 2.x 的模块（单位、仓库、公司）都是**扁平列表**。Story 3.1 是**树形结构**，有以下关键差异：

1. **自引用 FK**：`parent_id` 指向同表的 `id`，一级分类 `parent_id = NULL`
2. **level 字段**：由后端自动计算（parent 为 null → level=1；parent.level=1 → level=2；以此类推）
3. **sortOrder 字段**：`INT DEFAULT 0`，用于同级节点排序（当前按创建顺序，后续可扩展）
4. **树形 API**：`GET /product-categories/tree` 返回嵌套结构供前端 Tree 组件使用
5. **前端双面板布局**：左侧 Ant Design `Tree` + 右侧列表/详情，不是标准 ProTable 列表

#### Repository

```typescript
// product-categories.repository.ts
// 核心方法：

// 查询完整树（一次性拉取全部节点，在内存中组装为嵌套结构）
async findTree(): Promise<ProductCategoryNode[]> {
  const all = await this.repo.find({ order: { sortOrder: 'ASC', createdAt: 'ASC' } });
  return this.buildTree(all);
}

// 在内存中构建嵌套树（避免 N+1 查询）
private buildTree(items: ProductCategory[], parentId: number | null = null): ProductCategoryNode[] {
  return items
    .filter(item => item.parentId === parentId)
    .map(item => ({
      ...item,
      children: this.buildTree(items, item.id),
    }));
}
```

#### Service 层级校验逻辑

```typescript
// product-categories.service.ts

async create(dto: CreateProductCategoryDto, operator?: string) {
  // 1. 名称唯一性校验（全局唯一，简化设计）
  const dup = await this.repo.findByName(dto.name);
  if (dup) throw new BadRequestException('分类名称已存在');

  // 2. 计算 level
  let level = 1;
  if (dto.parentId) {
    const parent = await this.repo.findById(dto.parentId);
    if (!parent) throw new NotFoundException('父级分类不存在');
    if (parent.level >= 3) throw new BadRequestException('已达最大层级（3级），不能继续添加子分类');
    level = parent.level + 1;
  }

  return this.repo.create({ name: dto.name, parentId: dto.parentId ?? null, level, sortOrder: 0, createdBy: operator, updatedBy: operator });
}

async update(id: number, dto: UpdateProductCategoryDto, operator?: string) {
  const cat = await this.repo.findById(id);
  if (!cat) throw new NotFoundException('产品分类不存在');
  // 只允许修改名称（level/parent_id 不变）
  if (dto.name && dto.name !== cat.name) {
    const dup = await this.repo.findByName(dto.name);
    if (dup && dup.id !== id) throw new BadRequestException('分类名称已存在');
  }
  return this.repo.update(id, { name: dto.name, updatedBy: operator });
}
```

**树节点转换**：API 返回的 `ProductCategoryNode[]` 需转换为 antd Tree 要求的 `{title, key, children, disabled?, selectable?}` 格式。

**层级路径计算**：选中节点时，通过遍历树找到祖先路径（如"灯具 / 户外灯"）。

**三级禁用规则**：
- 右侧面板"新建子分类"按钮：当 `selectedNode.level >= 3` 时 `disabled`
- 表单中父分类选择（TreeSelect）：`level >= 3` 的节点设为不可选（`disableCheckbox: true` 或 `selectable: false`）

### 禁止事项（Anti-Patterns）

- **禁止** 使用 TypeORM 的 `@Tree` 装饰器或 `TreeRepository`（项目未使用此特性，在内存中手动组装树）
- **禁止** 在 Controller 中直接访问 Repository（必须通过 Service 层）
- **禁止** 在 Entity 中省略 `@Column({ name: 'snake_case' })` 和 `@Expose()`
- **禁止** 将 `level` 由前端传入（必须由 Service 根据 parent 自动计算）
- **禁止** 允许修改分类的 `parentId`（本 Story 只支持改名）
- **禁止** 在前端组件中直接 import axios（必须通过 `src/api/product-categories.api.ts` 层）
- **禁止** 使用 `useState` 管理服务端数据（必须 TanStack Query）
- **禁止** 在 `GET /tree` 端点使用分页（树必须一次性返回全部数据）

### 历史 Story 关键经验

来自 Story 2.4 的经验：
- `companies` 的 `findAll` 未使用 `applyPagination`/`applyKeywordFilter` 工具函数（内联实现）。新模块**可以使用**工具函数，也可以像 companies 那样内联（风格一致即可）
- `BaseQueryDto` 的实际文件名是 `base-query.dto.ts`（不是 `pagination.dto.ts`）
- `companies/index.tsx` 中仍有内联的 `useDebouncedValue`（未使用 Story 2.4 提取的 hook）。新模块**应使用** `import { useDebouncedValue } from '../../../hooks/useDebounce'`
- 后端 TypeScript 编译错误会导致测试失败，提前确认 tsc 无报错

来自 Story 2.3 的经验（公司主体）：
- `App.tsx` 是手动路由注册（非 lazy loading），直接 import 组件后添加 `<Route>`

## 前端 UX 规范

### 列表/树页面（UX-DR04 变体）

产品分类页面**不使用标准 ProTable 列表布局**，而使用**Tree + 右侧详情双面板**：

| 元素 | 要求 |
|------|------|
| 左侧 Tree 面板 | antd Tree，defaultExpandAll，选中节点高亮；顶部"新建一级分类"按钮（Primary） |
| 右侧详情面板 | 选中节点后显示：名称、层级路径（面包屑样式）、下属 SPU 数量（0）；按钮："编辑"（Secondary）、"新建子分类"（Dashed，三级时 disabled） |
| 未选中状态 | 右侧显示 Empty + "请选择左侧分类节点" |
| 加载状态 | Skeleton 骨架屏（UX-DR21） |
| 树为空 | Empty + "暂无分类" + "新建一级分类"按钮（UX-DR22） |

### 表单页（UX-DR06）

| 元素 | 要求 |
|------|------|
| 布局 | 简单表单（<10 字段）：单卡片，最多 2 字段 |
| 分类名称 | 必填，红色星号 |
| 父级分类 | 可选，antd TreeSelect；三级节点设为不可选；编辑时隐藏此字段 |
| 二次确认 | 无需（创建/保存不需要确认，UX-DR16） |
| 提交反馈 | message.success 3 秒（UX-DR17） |

### 状态与颜色
