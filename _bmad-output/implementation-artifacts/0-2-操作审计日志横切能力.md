# Story 0.2: 操作审计日志横切能力

Status: review

## Story

As a 系统管理员,
I want 系统自动记录所有数据变更操作（创建/更新/删除），包含操作人、时间、操作对象和变更内容,
so that 出现数据异常时可以追溯操作历史，满足业务合规要求。

---

## Acceptance Criteria

**Given** 授权用户执行任意 POST/PATCH/DELETE 请求  
**When** 请求成功完成（2xx 响应）  
**Then** `operation_logs` 表新增一条记录，包含：操作人用户名、操作时间、HTTP 方法、资源路径、操作类型（CREATE/UPDATE/DELETE）、请求体摘要（敏感字段脱敏）

**Given** 请求失败（4xx/5xx）  
**When** 操作未成功  
**Then** 不写入操作日志（只记录成功操作）

**Given** 管理员访问 `GET /api/v1/operation-logs`  
**When** 查询操作日志列表  
**Then** 返回分页列表，支持按操作人、操作类型、时间范围筛选  
**And** 按操作时间倒序排列

**Given** 前端操作日志列表页  
**When** 页面加载  
**Then** ProTable 展示：操作人、操作时间、操作类型 Tag、资源路径、请求摘要  
**And** 支持时间范围筛选和操作类型筛选

**Given** 详情页底部操作记录区（UX-DR13 ActivityTimeline）  
**When** 查看任意业务单据详情  
**Then** 展示该记录的操作历史时间线（按 resource_path + resource_id 关联查询）

## Dev Notes

### 关键设计决策

**OperationLog 不继承 BaseEntity**：BaseEntity 含 `created_by/updated_by` 字段，操作日志本身就是审计记录，不需要再被审计，避免递归写入。

**异步写入防止阻塞**：`AuditInterceptor` 使用 `tap()` 操作符在响应流完成后异步写入，写入失败不影响主请求响应。

**循环依赖处理**：`AuditInterceptor` 依赖 `OperationLogsService`，而 `OperationLogsModule` 需要在 `AppModule` 中注册。解决方案：`OperationLogsModule` 导出 `OperationLogsService`，`AppModule` 导入 `OperationLogsModule`，`AuditInterceptor` 通过构造函数注入。

**Migration 时间戳**：使用 `20260425000100`，当前最新为 `20260424000300`，无冲突。

### AuditInterceptor 核心逻辑

```typescript
import { tap } from 'rxjs/operators';

intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
  const req = context.switchToHttp().getRequest();
  const method = req.method;
  if (!['POST', 'PATCH', 'DELETE'].includes(method)) {
    return next.handle();
  }

  return next.handle().pipe(
    tap({
      next: () => {
        const res = context.switchToHttp().getResponse();
        if (res.statusCode < 400) {
          // 异步写入，不 await
          this.operationLogsService.create({...}).catch(err =>
            this.logger.warn('audit write failed', err)
          );
        }
      },
    }),
  );
}
```
