# Story 4.4: 客户主数据管理

Status: done

## Story

As a 授权用户,
I want 创建和管理客户主数据，记录客户代码、客户名称、客户国家、销售员及联系信息,
so that 销售员创建订单时可以直接选择客户，客户信息和归属销售员都有唯一数据来源。

---

## Acceptance Criteria

**Given** 授权用户进入客户列表页  
**When** 页面加载完成  
**Then** 展示客户列表（ProTable），含列：客户代码、客户名称、联系人、国家、销售员、创建时间  
**And** 支持按国家筛选、关键字搜索、分页

**Given** 授权用户点击"新建客户"  
**When** 进入客户创建表单  
**Then** 表单包含：客户代码（必填）、客户名称（必填）、客户国家（必填，关联国家/地区主数据）、销售员（必填，关联用户管理列表）、联系人、联系电话、联系邮箱、开票需求（多行文本）、地址  
**And** 选择国家后自动填充国家名称  
**And** 选择销售员后自动填充销售员姓名

**Given** 授权用户查看客户详情页  
**When** 页面加载完成  
**Then** ProDescriptions 展示所有客户字段  
**And** 展示该客户关联的销售订单列表（预留，Epic 5 落地）

**Given** 授权用户编辑客户信息  
**When** 修改联系人并保存  
**Then** 客户记录更新成功

**Given** 后端客户实体  
**When** 查看设计  
**Then** 包含 `id`、`customer_code`、`customer_name`、`country_id`、`country_name`、`salesperson_id`、`salesperson_name`、`contact_person`、`contact_phone`、`contact_email`、`billing_requirements`、`address`、`created_at`、`updated_at`、`created_by`、`updated_by`

## Dev Notes

### 业务与范围边界

- 本 Story 对应 FR12，仅覆盖客户主数据 CRUD，不包含销售订单真实关联实现；详情页中的销售订单区域仅做预留展示。[Source: `_bmad-output/planning-artifacts/epics.md` Epic 4 / Story 4.4]
- 客户是 Epic 5 销售订单的前置主数据，字段设计要兼顾后续订单表单的客户选择与国家展示需求。[Source: `_bmad-output/planning-artifacts/prd.md` FR12 / FR21]
- 你已确认当前 Story 在保留联系人/联系电话/联系邮箱/开票需求/地址字段的同时，新增 `客户代码` 与必填 `销售员` 字段，因此实现需以“核心主字段 + 扩展联系信息”组合落地。

### 后端实现约束

- 模块放在 `apps/api/src/modules/master-data/customers/`，遵守 7 文件标准结构。[Source: `_bmad-output/project-context.md` 后端实现规则]
- Entity 必须继承 `BaseEntity`，每个字段同时使用 `@Expose()` 与显式 `@Column({ name: 'snake_case' })`。[Source: `_bmad-output/project-context.md` Entity 规则]
- 当前主数据模块的实际返回结构使用 `list` 而不是文档中的 `data`，新增 customers 模块应与现有代码保持一致，避免前端 API 层额外适配。[Source: `apps/api/src/modules/master-data/companies/companies.repository.ts`, `apps/api/src/modules/master-data/countries/countries.repository.ts`]
- `countryName` 建议冗余存储，和 `company` 模块现有 `countryId/countryName` 模式保持一致，减少列表与详情展示时的额外 join 依赖。[Source: `apps/api/src/modules/master-data/companies/entities/company.entity.ts`]
- `salespersonName` 也建议冗余存储，原因与 `countryName` 一致，便于列表和详情直接展示。
- 创建/更新时应通过 `CountriesRepository` 或 `CountriesService` 校验 `countryId`，并同步写入 `countryName`，避免脏数据进入后续销售订单流程。[Inference based on existing company pattern + Story 4.4 AC]
- 搜索与分页沿用已有 QueryDto / QueryBuilder 模式；如需复用公共查询能力，优先参考 Story 2.4 的公共搜索能力，而不是重新发明一套查询接口。[Source: `_bmad-output/implementation-artifacts/2-4-搜索筛选与分页横切能力.md`]

### 前端实现约束

- 列表页、详情页、表单页分别放在 `apps/web/src/pages/master-data/customers/` 下，延续 `companies` / `product-documents` 模块组织方式。[Source: `_bmad-output/project-context.md` 前端实现规则]
- 禁止在页面组件中直接调用 axios，必须通过 `apps/web/src/api/customers.api.ts` 封装请求，并使用 TanStack Query 管理服务端状态。[Source: `_bmad-output/project-context.md` API 调用规则]
- 现有代码库尚未发现可复用的 `EntitySearchSelect` 组件实现；实际已落地模式是 `ProFormSelect` + `request` 远程搜索，因此国家选择建议优先遵循当前代码库模式，除非实现共享组件被明确纳入本 Story 范围。[Source: `_bmad-output/project-context.md` 组件选型规则, `apps/web/src/pages/master-data/companies/form.tsx`, repository scan]
- 列表页筛选交互已使用 `SearchForm` 组件，支持搜索、Tag 展示、重置等基础能力；客户列表应复用它并增加国家筛选项。[Source: `apps/web/src/components/common/SearchForm.tsx`, `apps/web/src/pages/master-data/companies/index.tsx`]
- 销售员关联可复用当前 `companies/form.tsx` 中的远程 `ProFormSelect` 用户搜索模式，直接调用 `/users?search=`，不必额外引入新组件。[Source: `apps/web/src/pages/master-data/companies/form.tsx`, `apps/web/src/api/users.api.ts`]
- Sidebar 已经存在 `/master-data/customers` 菜单项，但 `App.tsx` 与 `AppLayout.tsx` 还没有客户路由与面包屑映射，Story 实现时必须补齐，否则菜单点击会落空。[Source: `apps/web/src/components/layout/Sidebar.tsx`, `apps/web/src/App.tsx`, `apps/web/src/components/layout/AppLayout.tsx`]

### 数据模型建议

- 推荐实体命名：
  - `customerCode: string`
  - `customerName: string`
  - `contactPerson: string | null`
  - `contactPhone: string | null`
  - `contactEmail: string | null`
  - `countryId: number`
  - `countryName: string`
  - `salespersonId: number`
  - `salespersonName: string`
  - `billingRequirements: string | null`
  - `address: string | null`
- 推荐索引：
  - `customer_code` 唯一索引
  - `customer_name` 普通索引，满足 NFR-P3 搜索要求
  - `country_id` 普通索引，用于列表筛选
  - `salesperson_id` 普通索引，方便后续按销售员统计或过滤
- 如未添加数据库 FK，也需在服务层校验国家存在，保持与现有公司主体模块“存冗余展示字段 + 服务层校验”的风格一致。
