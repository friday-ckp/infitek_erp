# Story 4.2: 合同条款范本管理

Status: done

## Story

As a 授权用户,
I want 创建和管理带手动审批流的合同条款范本，维护模板名称、模板文件、模板说明、合同条款及模板属性，并手动推进审核状态,
So that 采购订单后续可以引用标准化且经过审核通过的合同条款，确保合同内容一致性。

> **字段清单约束**：根据 `_bmad-output/planning-artifacts/sprint-change-proposal-2026-04-20.md`，
> Epic 4 之后所有 Story 创建时必须对照 `docs/系统模块对应字段清单.md`。
> 本 Story 对照模块：`合同条款范本`
> （FormUUID: `FORM-E890D3E10BB440309E89D36D7FCA8DD4OJYB`）。

## Acceptance Criteria

**Given** 授权用户进入合同条款范本列表页
**When** 页面加载完成
**Then** 展示条款列表（ProTable），含列：合同模板名称、是否默认模板、是否为法务审核模板、状态、创建时间、更新时间
**And** 支持按状态筛选、关键字搜索、分页

**Given** 授权用户点击“新建条款”
**When** 填写合同模板名称（必填）、上传合同文件模板（文件类型，选填）、填写模板说明（选填）、填写合同条款（必填，多行文本）、选择是否默认模板、选择是否为法务审核模板
**Then** 条款创建成功，默认状态为“待提交”
**And** 详情页可正确展示模板文件信息、模板说明、合同条款和模板属性

**Given** 条款处于“待提交”状态
**When** 授权用户点击“提交审核”
**Then** 系统弹出 Popconfirm 确认（UX-DR16 状态推进）
**And** 确认后状态变更为“审核中”

**Given** 条款处于“审核中”状态
**When** 授权用户点击“审核通过”或“已拒绝”
**Then** 系统弹出 Popconfirm 确认
**And** 确认后状态分别变更为“审核通过”或“已拒绝”

**Given** 条款处于“待提交”“审核中”“已拒绝”或“审核通过”状态
**When** 授权用户点击“作废”
**Then** 系统弹出 Popconfirm 确认
**And** 确认后状态变更为“已作废”

**Given** 条款处于“审核通过”状态
**When** 授权用户编辑条款内容
**Then** 允许编辑并保存
**And** 状态保持“审核通过”不变

**Given** 一条“审核通过”状态的合同条款已被一张或多张“已确认”采购订单引用
**When** 授权用户编辑该条款内容并保存
**Then** 系统采用“引用时关联 ID”策略：已有采购订单的条款内容不受影响（关联的是条款 ID，详情页展示时实时拉取当前条款内容）
**And** 系统在编辑保存操作时展示提示：“该条款已被 N 张采购订单引用，修改后引用方将展示最新条款内容，请确认修改内容已获相关人员知悉。”（Popconfirm 二次确认）

**Given** 后端条款实体
**When** 查看设计
**Then** 包含 id、name、template_file_key、template_file_name、description、content、is_default、requires_legal_review、status、created_at、updated_at、created_by、updated_by

## Dev Agent Context

### 业务与字段边界

- 本 Story 属于供应链参考主数据，核心目标是提供可被采购订单引用的合同条款主数据源
- 采购订单对合同条款的实际引用发生在 Epic 6；当前 Story 只需保证“仅审核通过条款可被引用”的数据边界与可扩展接口
- PRD 明确“合同条款只在一处维护，下游所有单据通过关联引用获取”，因此本 Story 采用“引用时关联 ID”而不是“编辑时快照复制”
- Story 4.2 的 AC 已补充“被已确认采购订单引用后再次编辑”的一致性策略，后续采购订单详情应实时读取当前条款内容
- 审批流在本 Story 内先以系统内手动状态推进落地，不对接钉钉、飞书、OA 等第三方审批流

### 字段清单引用结论

`docs/系统模块对应字段清单.md` 中“合同条款范本”字段如下：

1. 合同模板名称
2. 合同文件模板
3. 模板说明
4. 合同条款
5. 范本ID
6. 是否默认模板
7. 状态
8. 流程是否完结

**本 Story 建议映射策略：**

- 纳入本 Story：
  - 1 合同模板名称 → `name`
  - 2 合同文件模板 → `templateFileKey` / `templateFileName`
  - 3 模板说明 → `description`
  - 4 合同条款 → `content`
  - 5 范本ID → `id`（系统主键，对外展示可直接复用记录 ID）
  - 6 是否默认模板 → `isDefault`
  - 7 状态 → `status`
- 额外补充字段：
  - 是否为法务审核模板 → `requiresLegalReview`
- 当前不单独落库的概念：
  - 8 流程是否完结：可由状态推导，`审核通过 / 已拒绝 / 已作废` 视为流程已结束，无需单独字段

### 状态枚举建议

为遵守“共享枚举定义在 `packages/shared`”规则，建议采用英文值 + 中文展示文案：

```typescript
export const ContractTemplateStatus = {
  PENDING_SUBMIT: 'pending_submit',
  IN_REVIEW: 'in_review',
  APPROVED: 'approved',
  REJECTED: 'rejected',
  VOIDED: 'voided',
} as const;
```

前端展示文案：

- `pending_submit` → 待提交
- `in_review` → 审核中
- `approved` → 审核通过
- `rejected` → 已拒绝
- `voided` → 已作废

### 手动审批流建议

- 新建后默认状态：`待提交`
- `待提交` -> `审核中`：提交审核
- `审核中` -> `审核通过`：审核通过
- `审核中` -> `已拒绝`：驳回
- `待提交 / 审核中 / 已拒绝 / 审核通过` -> `已作废`：作废
- 当前阶段不接入第三方审批系统，不保留审批实例号，仅保留业务状态和审计字段

### 后端实现护栏

- 模块目录遵循 7 文件标准：module / controller / service / repository / entity / dto / tests
- Entity 必须继承 `BaseEntity`
- 所有枚举定义在 `packages/shared`，前后端共同引用，禁止重复定义
- `status` 只能通过动作端点推进，通用更新接口不允许直接接收 `status`
- 文件模板建议复用现有文件上传方案：先上传文件获取 `key/filename`，再将文件元数据作为 JSON 字段提交到主数据接口
- Repository 列表查询需支持：
  - `keyword`：搜索 `name` / `description` / `content`
  - `status`
  - `page` / `pageSize`
- 为后续采购订单集成预留 `countApprovedUsages(id: number)` 或同等语义方法；若采购订单模块不存在，返回 `0`
- Migration 需显式创建索引，至少覆盖 `status`、`created_at`

### 实体建议

```typescript
@Entity('contract_templates')
@Unique('uq_contract_templates_name', ['name'])
@Index('idx_contract_templates_status', ['status'])
export class ContractTemplate extends BaseEntity {
  @Column({ name: 'name', type: 'varchar', length: 200 })
  @Expose()
  name: string;

  @Column({ name: 'template_file_key', type: 'varchar', length: 500, nullable: true })
  @Expose()
  templateFileKey: string | null;

  @Column({ name: 'template_file_name', type: 'varchar', length: 200, nullable: true })
  @Expose()
  templateFileName: string | null;

  @Column({ name: 'description', type: 'text', nullable: true })
  @Expose()
  description: string | null;

  @Column({ name: 'content', type: 'longtext' })
  @Expose()
  content: string;

  @Column({ name: 'is_default', type: 'tinyint', width: 1, default: () => '0' })
  @Expose()
  isDefault: boolean;

  @Column({ name: 'requires_legal_review', type: 'tinyint', width: 1, default: () => '0' })
  @Expose()
  requiresLegalReview: boolean;

  @Column({ name: 'status', type: 'varchar', length: 20 })
  @Expose()
  status: ContractTemplateStatus;
}
```

### 前端实现护栏

- 列表页使用 `ProTable`，优先参考：
  - `apps/web/src/pages/master-data/companies/index.tsx`
  - `apps/web/src/pages/master-data/certificates/index.tsx`
  - `apps/web/src/pages/master-data/product-documents/index.tsx`

- 详情页遵循升级后的详情规范：摘要卡片 + 基本信息 Tab + 使用说明/引用提示区
- 状态标签遵循 UX-DR07：
  - `pending_submit` → 灰色 `default`
  - `in_review` → 橙色 `warning`
  - `approved` → 绿色 `success`
  - `rejected` → 红色 `error`
  - `voided` → 红色 `error`
- 编辑“审核通过”条款时：
  - 若引用数为 `0`，直接保存
  - 若引用数 `> 0`，先展示 Popconfirm，再执行保存
- 列表页必须包含：
  - 页面标题 + “新建条款”主按钮
  - 关键字搜索（300ms debounce）
  - 状态筛选
  - 已选筛选条件标签
- 详情页动作按钮建议：
  - `待提交`：编辑 / 提交审核 / 作废
  - `审核中`：审核通过 / 已拒绝 / 作废
  - `审核通过`：编辑 / 作废
  - `已拒绝`：编辑 / 提交审核 / 作废
  - `已作废`：仅查看

### 与后续 Story 的关系

- Story 4.1 中字段清单里的“合同范本 / 合同条款 / 范本ID”不在 4.1 MVP 内，本 Story 提供其主数据来源
- Epic 6 的采购订单创建需要“默认关联供应商下状态为已审批的第一条合同条款范本”；本 Story 应确保存在按 `status=approved` 查询能力
- 当前 Story 不要求实现供应商与合同条款绑定关系；如产品代码中尚无 `suppliers` 模块，不应阻塞本 Story 开发
