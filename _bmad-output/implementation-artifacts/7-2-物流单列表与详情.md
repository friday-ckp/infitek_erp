# Story 7.2: 物流单列表与详情

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 授权用户,
I want 查看物流单列表（按状态和物流供应商筛选）和物流单详情，
so that 我可以跟踪物流进度并查看货物、物流跟踪和关联单据。

## Scope Adjustment

- Story 7.1 已完成物流单创建、创建即确认、物流跟踪字段预留和基础查询接口；本 Story 不从零重建 logistics-orders 模块，而是在现有后端 `findAll/findById` 与前端创建页基础上补齐正式列表页、详情页和物流跟踪后补能力。
- 本 Story 聚焦三件事：物流单列表页、物流单详情页、物流跟踪字段详情页内联编辑。
- 本 Story 不实现发货出库单创建与确认、不推进物流单到 `shipped/delivered`、不扣减库存、不实现物流费用落地；这些属于 Story 7.3 / 7.4 / 7.5。
- Epic 文本中的“装柜方式”当前在现有实体、旧系统字段与 Story 7.1 落地模型里没有稳定字段来源；本 Story 先以可直接落地的“运输方式 + 装箱信息摘要”承接列表跟踪需求，若你要求强行补“柜型/装柜方式”独立字段，则需要同步补 migration、创建页和详情页模型。

## Acceptance Criteria

1. 授权用户进入 `/logistics-orders` 后，展示正式物流单列表页（ProTable），至少包含：物流单编号（蓝色链接）、关联发货需求编号（蓝色链接）、物流供应商、起运港/目的港、状态（彩色 Tag）、创建时间。
2. 列表支持关键字搜索、状态筛选、物流供应商筛选和分页；关键字至少覆盖物流单编号、发货需求编号、销售订单号、客户名称。
3. 列表页遵守 UX-DR04：页面标题、搜索 + 高级筛选、已选条件 Tag、固定表头、固定右侧操作列、底部分页器与总数。
4. 点击物流单编号或“查看”进入 `/logistics-orders/:id`；详情页按 UX-DR05 展示摘要卡片、锚点导航、信息分区和状态 pill。
5. 物流单详情页至少展示以下分区：
   - 基本信息：物流供应商、运输方式、公司主体、起运港、目的港、运抵国、要求到货日期。
   - 物流跟踪：ETD、ETA、订舱号、船司/航司、船名航次、SO/提单号、实际离港日期；空值展示 `—`。
   - 货物明细：SKU、产品名称、计划数量、已出库数量、单位。
   - 装箱信息：SKU、箱号、箱数、每箱数量、总数量、尺寸、毛重。
   - 报关信息：是否出口报关、唛头。
   - 关联单据：发货需求链接、销售订单链接、出库单链接占位说明。
6. 详情页使用 `ActivityTimeline` 展示至少“创建物流单”和后续“物流跟踪信息更新”等事件；不得伪造尚未发生的出库事件。
7. 当物流单状态为 `confirmed` 或 `shipped` 时，用户可在详情页点击“编辑物流跟踪”进入内联编辑，允许更新 ETD、ETA、订舱号、船司/航司、船名航次、SO/提单号、实际离港日期；保存后刷新详情并新增操作记录。
8. 物流跟踪更新必须走专用后端动作接口，禁止通用 PATCH 任意改状态；本 Story 只允许编辑物流跟踪字段，不允许直接修改 `status`、数量或上游单据关系。
9. 后端列表/详情/物流跟踪更新实现必须继续遵守 Epic 5-7 flow 约束：不新增库存写操作、不将待采购数量写入物流单、不绕过 shared 枚举、不通过详情编辑直接推进交易状态。
10. 补充后端测试覆盖物流单列表筛选、详情读取、物流跟踪更新校验；前端至少通过 `web build`，并完成 API 类型、路由和详情交互验证。
