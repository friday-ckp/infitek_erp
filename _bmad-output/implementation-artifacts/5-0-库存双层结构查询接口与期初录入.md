# Story 5.0: 库存双层结构查询接口与期初录入

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 系统管理员和开发者,
I want 在生成发货需求前先建立统一的交易状态枚举、库存双层结构、期初库存录入和可用库存查询接口,
so that Story 5.2 可以基于正式库存数据生成发货需求，避免临时库存方案和后续返工。

## Acceptance Criteria

1. 共享枚举补齐并导出：
   `ShippingDemandStatus` 使用 `as const` 对象模式定义 `pending_allocation`、`purchasing`、`prepared`、`partially_shipped`、`shipped`、`voided`；新增 `FulfillmentType`：`full_purchase`、`partial_purchase`、`use_stock`；新增 `InventoryBatchSourceType`：`initial`、`purchase_receipt`；新增 `InventoryChangeType`：`initial`、`purchase_receipt`、`outbound`、`lock`、`unlock`。所有新增枚举必须从 `packages/shared/src/index.ts` 导出，禁止前后端重复定义。
2. 后端新增正式 `InventoryModule`，包含标准模块结构、实体、仓储、服务、控制器、DTO、测试，并注册到 API 应用；不得实现临时库存表、mock API 或硬编码库存快照字段。
3. 数据库迁移创建汇总层 `inventory_summary`：`id`、`sku_id`、`warehouse_id`、`actual_quantity`、`locked_quantity`、`available_quantity`、`created_at`、`updated_at`、`created_by`、`updated_by`，并对 `sku_id + warehouse_id` 建唯一约束；`available_quantity` 由 Service 层维护，不使用数据库计算列。
4. 数据库迁移创建批次层 `inventory_batch`：`id`、`sku_id`、`warehouse_id`、`batch_quantity`、`batch_locked_quantity`、`source_type`、`source_document_id`、`receipt_date`、`created_at`、`updated_at`、`created_by`、`updated_by`，并建立 SKU、仓库、FIFO 查询所需索引；汇总层与批次层数量必须保持一致。
5. 期初库存录入能力可由系统管理员选择 SKU、仓库、期初数量、入库日期并提交；提交后创建或覆盖该 SKU+仓库的 `source_type=initial` 期初批次，并同步汇总层 `actual_quantity`、`locked_quantity`、`available_quantity`。
6. 库存查询接口 `GET /api/inventory/available?skuIds=1,2,3` 返回每个 SKU 在各仓库的实际库存、锁定量、可用库存：`[{ skuId, warehouseId, actualQuantity, lockedQuantity, availableQuantity }]`；可选 `warehouseId` 参数限定单个仓库；某 SKU+仓库无汇总记录时返回 `availableQuantity=0`，不报错；目标响应时间 < 1 秒。
7. 后续库存写操作入口必须集中在 `InventoryService`，以 TypeORM `QueryRunner` 事务、固定锁顺序和 `SELECT ... FOR UPDATE` 实现锁定/解锁/扣减/增加；死锁指数退避最多 3 次，超限抛 `ConflictException({ code: 'CONCURRENT_UPDATE' })`。
8. 本 Story 必须提供后端定向单测，覆盖期初录入创建/覆盖、可用库存查询、无库存返回 0、并发保护方法调用/死锁重试边界；构建和选定单测通过。
9. 前端新增库存基础入口，可完成期初库存录入并查看 SKU+仓库库存查询结果；API 调用必须通过 `src/api/` 层和 TanStack Query，不在组件内直接调用 axios。
10. Story 5.2 在本 Story 完成前不得实现生成发货需求；5.2 后续必须消费本 Story 的正式库存查询接口。

### Data Model Notes

建议字段类型：

- `inventory_summary.sku_id`、`warehouse_id`：`bigint unsigned`，FK 到 `skus.id`、`warehouses.id`。
- 数量字段：优先使用 `int`，与销售订单 `quantity` 当前口径一致；如实现时判断需要小数，必须同步说明并保证前后端类型一致。
- `inventory_summary` 唯一键：`uq_inventory_summary_sku_warehouse(sku_id, warehouse_id)`。
- `inventory_batch.source_type`：`varchar(40)` 或 enum；若使用 DB enum，值必须来自 shared 枚举，避免中文持久化值。
- `inventory_batch.source_document_id`：期初录入可为空；采购入库后续填收货/采购来源。
- `receipt_date` 用于 FIFO，索引建议包含 `sku_id, warehouse_id, receipt_date, id`。

期初覆盖口径：

- 同一 SKU+仓库只允许一个 `source_type=initial` 的期初批次；再次提交期初数量时更新该批次数量和 receiptDate，而不是追加第二条 initial batch。
- 覆盖后必须从该 SKU+仓库全部批次聚合 summary，避免未来采购批次被错误清零。
- 若 initial batch 已存在锁定量，覆盖不得导致 `batch_quantity < batch_locked_quantity`；应返回业务错误，提示当前已锁定量。

### API Contract

后端建议接口：

- `POST /api/inventory/opening-balances`
  - request: `{ skuId: number, warehouseId: number, quantity: number, receiptDate?: string }`
  - response: `{ summary, batch }` 或更新后的 summary DTO。
- `GET /api/inventory/available?skuIds=1,2,3&warehouseId=10`
  - response item: `{ skuId: number, warehouseId: number | null, actualQuantity: number, lockedQuantity: number, availableQuantity: number }`

错误口径：

- SKU 或仓库不存在：`NotFoundException`，中文消息清晰。
- 数量为负数或非整数：`BadRequestException`。
- 覆盖期初导致数量小于已锁定量：`BadRequestException({ code: 'STOCK_INSUFFICIENT', available: N })` 或项目现有异常包装等价实现。
- 死锁重试超限：`ConflictException({ code: 'CONCURRENT_UPDATE' })`。
