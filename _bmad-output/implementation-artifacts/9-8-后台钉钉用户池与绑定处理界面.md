# Story 9.8: 后台钉钉用户池与绑定处理界面

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 系统管理员,
I want 在后台查看同步结果、匹配候选和冲突状态，并完成绑定处理,
so that 我可以在一个清晰的工作区内完成钉钉身份运营。

## Acceptance Criteria

1. **钉钉用户池工作区可见且可筛选**
   - 管理员进入钉钉用户同步与绑定管理页面时，可看到钉钉用户池记录、状态、匹配建议、最近同步时间
   - 支持按状态筛选和关键词搜索
   - 页面展示与现有后台风格一致，适合作为管理员运营工作区持续使用

2. **候选记录可直接确认绑定**
   - 当记录状态为 `CANDIDATE` 时，管理员可以直接确认绑定
   - 系统完成正式绑定后，列表与详情状态更新为 `BOUND`
   - 正式绑定结果仍回写到 `users.dingtalk_*` 字段，而不是另建平行正式绑定模型

3. **冲突或未绑定记录可手动选择 ERP 用户后绑定**
   - 当记录状态为 `CONFLICT` 或 `UNBOUND` 时，管理员可手动选择目标 ERP 用户并确认绑定
   - 系统保存正式绑定结果并给出明确成功反馈
   - 不自动开户，不在无管理员确认时自动绑定

4. **已绑定记录可解绑并回到待处理态**
   - 管理员对已绑定关系执行解绑并确认后，系统清除正式绑定关系
   - 用户池状态回到待处理态，并保留后续继续处理所需的候选池信息

## Dev Notes

### Story Boundary

- 本 Story 的目标是交付一个独立、可操作的“钉钉用户同步与绑定管理”后台工作区，而不是继续扩张现有 `用户管理` 页面。
- 本 Story 不重新设计匹配建议算法；手机号/邮箱/工号唯一命中、冲突判定等规则继续服从 Epic 9 已定义的匹配语义。
- 本 Story 不引入自动开户，不允许未确认数据直接进入扫码登录主链路。
- 本 Story 的正式绑定结果仍然只认 `users` 表中的 `dingtalk_*` 字段；候选池只是后台运营面。

### Epic And Story Context

- Epic 9 的目标不是只做扫码登录，而是补齐“组织同步 -> 候选匹配 -> 管理员确认绑定 -> 扫码登录”的完整运营闭环。[Source: `_bmad-output/planning-artifacts/epics.md#Epic 9`]
- Story 9.6 已经实现钉钉组织用户池表、同步接口、列表接口与基础前端占位区，但当前仅支持“同步 + 查询”，没有独立运营工作区，也没有正式的池内绑定处理流。
- Story 9.7 定义了匹配建议与冲突状态引擎：唯一命中为 `CANDIDATE`，多命中/冲突为 `CONFLICT`，未命中为 `UNBOUND`，管理员确认后才写回 `users` 表。[Source: `_bmad-output/planning-artifacts/epics.md#Story 9.7`]
- Story 9.8 是把这些后端状态真正变成可运营后台界面的故事，因此实现时必须明确区分“算法与状态来源”以及“工作区交互层”。

### Previous Story Intelligence

- `9-6-钉钉组织用户同步与用户池建模.md` 已明确禁止把候选池逻辑揉进 `AuthService` 或 `UsersService` 通用 CRUD；9.8 继续沿用 `apps/api/src/modules/dingtalk-org-users/` 作为后台运营聚合点。
- 9.6 已明确临时管理员边界仍是 `currentUser.username === 'admin'`；本 Story 不能虚构角色系统或把它换成不存在的 RBAC。
- 9.6 已要求候选池只作为后台运营数据，不直接参与扫码登录放行；9.8 不得为了图省事在登录链路上查询 `dingtalk_org_users`。
- 当前仓库已经把“钉钉组织用户池”临时嵌在 `apps/web/src/pages/settings/users/index.tsx` 中；9.8 应把这段占位实现迁移或重构为独立工作区，而不是继续把复杂交互压在用户列表页。

### Current Code Reality

- 后端已存在 `apps/api/src/modules/dingtalk-org-users/`：
  - `dingtalk-org-users.controller.ts` 当前只有 `GET /dingtalk-org-users` 与 `POST /dingtalk-org-users/sync`
  - `dingtalk-org-users.service.ts` 当前只有 `findAll()` 与 `sync()`
  - `dingtalk-org-users.repository.ts` 已支持分页、状态筛选和关键词搜索
  - `entities/dingtalk-org-user.entity.ts` / `20260601000100-create-dingtalk-org-users-table.ts` 已提供 `status`、`suggestedUserId`、`matchReason`、`lastSyncedAt`、`lastProcessedAt`、`processedBy`
- 正式绑定能力已存在于 `apps/api/src/modules/users/users.service.ts`：
  - `bindDingtalkIdentity()` 负责唯一性校验并回写 `users.dingtalk_*`
  - `unbindDingtalkIdentity()` 负责清空正式绑定字段
- 前端当前事实：
  - `apps/web/src/pages/settings/users/index.tsx` 已有同步按钮和候选池占位表格
  - `apps/web/src/pages/settings/users/detail.tsx` 已有用户级别的兜底绑定/解绑操作
  - `apps/web/src/api/dingtalk-org-users.api.ts` 目前只有列表与同步 API，9.8 需要为绑定处理动作扩展 API 客户端

### Architecture Compliance

- 当前真实 API 前缀是 `/api`，不是旧文档里的 `/api/v1`；新增接口和前端请求都必须按 `/api/...` 推导。[Source: `_bmad-output/project-context.md#Critical Don't-Miss Rules`]
- 统一响应格式继续遵循现有后端封装，不要在新接口中单独发明不兼容的返回结构。[Source: `_bmad-output/planning-artifacts/architecture.md#API 与通信模式`]
- 扫码登录主链路继续只依赖 `users.dingtalk_union_id` 查找正式绑定关系；候选池不能成为登录授权事实源。[Source: `_bmad-output/planning-artifacts/architecture.md#认证与安全`]
- 钉钉用户池至少要保留稳定身份键、昵称、手机号、邮箱、工号、部门、建议匹配用户、匹配依据、状态和时间戳；9.8 页面展示不能绕开这些现有字段自造临时结构。[Source: `_bmad-output/planning-artifacts/architecture.md#认证与安全`]

### Frontend Guidance

- 项目当前前端标准要求服务端数据优先使用 TanStack Query 的 `useQuery/useMutation`，不要继续在新页面里用 `useEffect + useState` 手写完整请求生命周期；如果沿用现有老模式，至少把 9.8 新增交互部分提升为 Query/Mutation 驱动，避免状态同步继续恶化。[Source: `_bmad-output/project-context.md#Frontend Framework Rules`]
- 新页面必须同步检查并更新 `App.tsx`、`Sidebar.tsx`、`AppLayout.tsx`，保证导航、路由和面包屑一致。[Source: `_bmad-output/project-context.md#Frontend Framework Rules`]
- UI 风格应延续现有 `master-page.css` 与 `SearchForm` 组件组合，不另起一套视觉体系；适合使用分组卡片 + 表格 + 弹窗/抽屉的后台运营模式。[Source: `_bmad-output/planning-artifacts/ux-design-specification.md#DingTalk Binding Operations`]
- UX 要点不是“把记录列出来”，而是让管理员能在一个工作区内快速看到状态、候选依据、冲突原因、最近同步与处理结果，减少手工查身份字段的焦虑。[Source: `_bmad-output/planning-artifacts/ux-design-specification.md#Core User Experience`]

### Backend Guidance

- 推荐在 `apps/api/src/modules/dingtalk-org-users/` 内新增 DTO / endpoint / service 方法，例如：
  - `POST /dingtalk-org-users/:id/confirm-binding`
  - `POST /dingtalk-org-users/:id/manual-binding`
  - `POST /dingtalk-org-users/:id/unbind`
- 以上命名可调整，但职责边界不要漂移到 `PATCH /users/:id` 之类的通用用户编辑接口中。
- 绑定动作内部应先读取对应候选池记录，再调用 `UsersService.bindDingtalkIdentity()` 完成正式绑定，随后回写池记录状态与处理信息。
- 解绑动作内部应调用 `UsersService.unbindDingtalkIdentity()`，然后根据记录现有候选信息恢复池状态，而不是直接软删用户池记录。
- 若实现中需要展示“建议 ERP 用户摘要”，优先通过后端 join/补充查询统一返回，不要求前端二次拼装多次请求瀑布流。

### Data And State Rules

- 状态枚举继续使用共享包中的 `DingtalkOrgUserStatus`：`UNBOUND`、`CANDIDATE`、`CONFLICT`、`BOUND`。[Source: `packages/shared/src/enums/dingtalk-org-user-status.enum.ts`]
- 不要新增“PROCESSING”“REBOUNDING”等未在 Epic/Architecture 中定义的中间状态，除非先同步更新共享枚举、架构文档和前后端逻辑。
- `suggestedUserId` / `matchReason` 是匹配引擎输出，不应由前端手工拼装或静态写死。
- 已绑定关系的唯一性仍受 `users.dingtalk_union_id` 约束；出现重复绑定时应延续既有错误码 `DINGTALK_IDENTITY_ALREADY_BOUND` 和统一错误语义。
