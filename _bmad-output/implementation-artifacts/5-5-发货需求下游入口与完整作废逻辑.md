# Story 5.5: 发货需求下游入口与完整作废逻辑

Status: done

## Story

As a 商务跟单,
I want 在发货需求详情页直接创建物流单，并对待分配库存状态的发货需求执行完整作废,
so that 我可以从枢纽页继续推进发货路径，或在作废后释放锁定库存、回退销售订单并重新生成干净的新发货需求。

## Scope Adjustment

- 本 Story 承接 Story 5.4 已完成的发货需求枢纽详情、履行类型确认、FIFO 锁定、批次级 allocation 和库存流水能力。
- 本 Story 必须完成发货需求作废闭环：后端专用动作端点、事务内释放 active allocation、写 unlock 库存流水、更新 allocation/明细/需求/销售订单状态、操作日志、前端二次确认和成功刷新。
- 本 Story 同时补齐“创建物流单”下游入口：从发货需求详情页跳转到物流单创建表单，自动带入当前发货需求、来源销售订单、客户、收发通、发货要求、可计划 SKU 与数量；预填字段以灰底只读展示。
- 当前代码尚无物流单业务模块。本 Story 可以新增物流单 MVP 创建表单与最小持久化接口，以满足 FR28/FR31/FR32 的入口和预填要求；发货出库单、出库确认、物流单发运/送达、物流费用不在本 Story 范围。
- 作废规则以三份 flow 文档为准：只允许 `pending_allocation` 作废；已离开待分配库存的需求不得作废，即使旧 PRD 有“处理中”旧口径也不沿用。

## Acceptance Criteria

1. 发货需求详情页在可创建物流单的状态展示“创建物流单”入口；点击后跳转物流单创建表单，并通过 URL 或状态参数带入当前发货需求 ID。
2. 物流单创建表单加载后从发货需求详情接口预填：发货需求编号、来源销售订单号、客户、运抵国、起运地、目的港/目的地、运输方式、签约公司、内外销、要求到货日期、收发通信息、唛头/开票/随货文件/清关要求等可继承字段。
3. 物流单产品明细只预填已锁定且尚未出库的 SKU 行；计划数量默认等于 `lockedRemainingQuantity`，且不可超过当前已锁定待发数量。预填的来源字段、SKU 和数量字段灰底只读，不允许用户手工改来源需求或扩大计划数量。
4. 物流单表单包含 MVP 必填字段：物流供应商、运输方式、公司主体、起运港/地、目的港/目的地、运抵国，以及装箱信息子行；提交后创建为 `confirmed` 状态，不保存草稿。
5. 发货需求详情页仅在 `pending_allocation` 状态展示“作废”Danger 按钮；`purchasing`、`prepared`、`partially_shipped`、`shipped`、`voided` 均不展示作废按钮。
6. 点击“作废”后使用 Modal 二次确认，文案包含：“确认作废此发货需求？作废后将释放已锁定库存，销售订单状态回退为‘审核通过’，操作不可撤销”。若后端返回有关联采购订单/物流单数量，Modal 或错误提示必须清楚说明需先手动处理。
7. 后端新增专用动作端点 `POST /api/shipping-demands/:id/void`，不新增通用状态 PATCH，不允许前端直接提交目标状态。
8. 后端作废仅允许 `pending_allocation`。已作废再次作废返回当前结果或业务幂等结果，不重复释放库存；其他状态返回业务错误并不改变库存或状态。
9. 作废事务按 flow 锁顺序锁定销售订单、发货需求、明细、active allocation、库存汇总与批次。若存在 active allocation，必须调用 `InventoryService.unlockStockInTransaction()` 释放锁定库存，不允许直接写 `inventory_summary` 或 `inventory_batch`。
10. 作废释放库存后，`shipping_demand_inventory_allocations` 对应 active 记录更新为 `released`，`releasedQuantity` 增加，`lockedQuantity` 清零，`shippedQuantity` 不变；对应 `shipping_demand_items.lockedRemainingQuantity` 从 allocation 聚合回填为 0。
11. 作废库存流水写入 `inventory_transactions`，`changeType = unlock`，`lockedQuantityDelta` 为负数，`availableQuantityDelta` 为正数，`sourceDocumentType = shipping_demand`，`sourceActionKey = shipping-demand:{id}:void` 或稳定派生键，避免重复流水。
12. 作废成功后发货需求状态变为 `voided`，并记录 `voidedAt` / `voidedBy`；关联销售订单状态回退为 `approved`，销售订单详情页重新展示“生成发货需求”按钮。
13. 作废成功后前端展示 `notification.success`，刷新发货需求详情、发货需求列表、销售订单详情和操作记录；作废后的发货需求详情状态为红色“已作废”，不再展示分配库存、创建物流单、作废按钮。
14. 销售订单详情页“重新生成发货需求”复用 Story 5.2 的生成逻辑；因已作废需求不再算 active demand，系统必须生成新的发货需求单并保留旧作废单据审计。
15. 并发死锁或重复键冲突按指数退避最多 3 次重试，超限返回 `ConflictException({ code: 'CONCURRENT_UPDATE' })`。

## Dev Notes

### Epic 5-7 Mandatory Flow Context

本 Story 属于 Epic 5-7 交易链路，开发前必须完整加载并优先遵守：

- `_bmad-output/planning-artifacts/flow-cross-document-trigger.md`
- `_bmad-output/planning-artifacts/flow-state-machine.md`
- `_bmad-output/planning-artifacts/flow-quantity-data-lineage.md`

若 PRD、Architecture、UX、Epic 或旧 Story 文本与上述文档冲突，以三份 flow 文档为准。本 Story 的具体约束：

- 发货需求作废流程：`pending_allocation` -> `voided`，释放已锁定库存，销售订单回退至 `approved`，随后允许重新生成新发货需求。
- 作废只允许 `pending_allocation`；旧文档中的“处理中/待发运”等旧口径不得用于放宽作废条件。
- 库存数量权威来源是 `inventory_summary`、`inventory_batch`、`shipping_demand_inventory_allocations` 和 `inventory_transactions`。
- 发货需求作废是数量变更动作，必须在 TypeORM QueryRunner 事务内完成，遵守锁顺序：销售订单 -> 发货需求 -> 当前动作单据/明细 -> inventory_summary -> inventory_batch -> allocation。
- 作废若释放库存，必须写 `unlock` 库存流水；旧发货需求必须保留为作废审计记录，不删除。
- 重新生成发货需求必须生成新单据，不能复活旧作废单据。

### Implementation Boundaries

- 不实现采购订单创建。
- 不实现发货出库单、出库确认、allocation 消费和出库扣减。
- 不实现物流费用管理。
- 不允许作废 `purchasing` 或之后状态。
- 不新增通用状态 PATCH。
- 不绕过 `InventoryService` 直接改库存汇总或批次。
- 不删除旧发货需求；作废记录用于审计和重新生成依据。
- 不把物流计划数量当库存事实；物流单创建不改变库存数量。
