# Story 5.2: 销售订单列表与详情

Status: done

## Story

As a 授权用户,
I want 在已完成的销售订单详情页查看完整订单字段，并在订单审核通过后手动生成发货需求,
so that 我可以从销售订单继续推进履约，同时避免重复录入或重复生成下游单据。

## Scope Adjustment

- 销售订单列表页和详情页主体能力在实际代码中已经完成，本 Story 不重复实现列表页或重写详情页。
- 本次只补齐：
  - 销售订单详情字段完整性，重点补产品明细已保存字段展示。
  - “审核通过后生成发货需求”按钮。
  - 后端 `shipping-demands` 实体、迁移、生成接口与服务。
  - 销售订单详情中的关联发货需求展示。
  - 最小只读发货需求详情路由，避免关联链接断链；完整发货需求列表页/枢纽详情仍留给后续 Story。

## Acceptance Criteria

1. 销售订单详情页完整展示已保存的产品明细字段，包括 SKU、产品名、规格、SPU、电参数、插头信息、数量、单价、金额、币种、单位、采购/库存相关数量、材质、重量体积和图片。
2. 销售订单状态为 `SalesOrderStatus.APPROVED` 且不存在未作废发货需求时，详情页操作区展示“生成发货需求”按钮。
3. 用户点击按钮后弹出二次确认，确认后调用后端生成接口。
4. 后端新增 `shipping_demands` 与 `shipping_demand_items` 实体和迁移，实体继承 `BaseEntity`，字段使用显式 snake_case。
5. 后端生成接口只允许从审核通过销售订单生成发货需求；同一销售订单存在未作废发货需求时拒绝重复生成。
6. 生成发货需求时包含销售订单所有 SKU 行及应发数量，调用 Story 5.0 的 `InventoryService.findAvailable()` 查询每个 SKU 当前可用库存并写入快照。
7. 发货需求初始状态为 `ShippingDemandStatus.PENDING_ALLOCATION`，行项初始履行类型为空。
8. 生成成功后，销售订单状态从 `SalesOrderStatus.APPROVED` 推进为 `SalesOrderStatus.PREPARING`。
9. 销售订单详情页展示关联发货需求编号、状态、创建时间；生成成功后隐藏生成按钮并展示成功通知。

### Epic 5-7 Mandatory Flow Context

本 Story 属于 Epic 5-7 交易链路，开发前必须完整加载并优先遵守：

- `_bmad-output/planning-artifacts/flow-cross-document-trigger.md`

若 PRD、Architecture、UX、Epic 或旧 Story 文本与上述文档冲突，以三份 flow 文档为准。本 Story 的具体约束：

- 销售订单审核通过后，由用户在详情页手动触发生成发货需求，不在审核通过时自动生成。
- 生成发货需求后销售订单状态推进到 `preparing`。
- 发货需求初始状态为 `pending_allocation`。
- 发货需求是履约数量权威单据；销售订单明细的履约数量字段只作为展示缓存或聚合结果。
- 禁止实现临时库存表、mock 库存 API 或硬编码可用库存；生成时必须消费 Story 5.0 的正式库存查询能力。
- 同一销售订单同一时间只允许存在一个非作废发货需求。

### Implementation Notes

- 销售订单列表页和详情页主体已经存在，避免重复实现或大面积重写。
- 后端模块路径采用 `apps/api/src/modules/shipping-demands/`。
- 前端详情页继续使用 `apps/web/src/pages/sales-orders/detail.tsx` 的现有 Summary Card + SectionCard 布局。
- `apps/web/src/pages/shipping-demands/detail.tsx` 只作为生成结果只读查看页，不实现 Story 5.3/5.4 的完整列表、库存分配和枢纽操作。
- 生成接口采用领域动作接口，不通过通用 PATCH 修改状态。
