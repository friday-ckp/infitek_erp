## 关键发现

### 亮点

1. **避坑指南文化成熟** — 3-1 记录了 NestJS 路由顺序陷阱（`GET /tree` 必须在 `GET :id` 前注册），3-5 和 3-6 均有详细"避坑指南（必读）"章节，知识沉淀质量明显高于 Epic 2。

2. **Story 间复用加速** — 3-6 产品资料库直接以 3-5 证书模块为参考，后端 7 文件结构几乎一比一复用，Completion Notes 显示实现顺畅，单测 16/16 通过。

3. **Code Review 有效拦截问题** — 3-2 的 review 发现并修复了 5 个 Patch 问题，包括 `generateCode` 字符串排序 bug（MAX 在序号 > 999 时失效）、`useDebouncedValue` 内联重复、`updateMutation` 类型错误等，质量门禁有效。

4. **树形结构方案稳健** — 3-1 明确禁止使用 TypeORM @Tree，采用手动内存组装树，与项目整体架构保持一致，且在 3-2（TreeSelect）、3-5（categoryId 展示）、3-6（分类层级联动）中被正确复用。

5. **OSS 集成模式固化** — 3-5 建立的两步式文件上传模式（先上传获取 key，再随表单提交）在 3-6 中被直接复用，FilesService @Global 注入方式也已成为团队共识。

## 技术规范沉淀（Epic 5 必须遵守）

### 后端

- 模块路径：`apps/api/src/modules/{domain}/{module}/`（Epic 5 的 domain = `sales`）
- Entity 继承 `BaseEntity`，无软删除（`repo.delete(id)` 直接物理删除）
- QueryDto 继承 `BaseQueryDto`，Repository 调用 `applyPagination` / `applyKeywordFilter`
- Migration 必须实现 `up()` + `down()`，时间戳唯一（Epic 5 从 `20260425000100` 起）
- Service 单元测试覆盖 CRUD 全路径
- FilesService 是 @Global()，直接注入，无需 import FilesModule
- ManyToMany 保存需传 Entity 对象数组（非 ID 数组）

### 前端

- 列表页：`ProTable` + `SearchForm` + `useDebounce`（强制）
- 详情页：`ProDescriptions` 分组卡片 + `ActivityTimeline`（Epic 5 新增）
- 表单页：`ProForm` + `ProCard` 分组；子表格行项用 ProTable editable 或自定义
- API 层：`import request from './request'`，所有函数 `.catch(normalizeApiError)`
- 服务端状态：TanStack Query，queryKey 格式 `[module, 'list', queryParams]`
- 分页类型：统一使用 `PaginatedResponse<T>`
- 日期字段：提交时 `.format('YYYY-MM-DD')`，回填时 `dayjs(detail.date)`
- ProFormDatePicker 提交值是 Dayjs 对象，需显式格式化
