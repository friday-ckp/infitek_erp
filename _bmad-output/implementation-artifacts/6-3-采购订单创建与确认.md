# Story 6.3: 采购订单创建与确认

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 采购员,
I want 创建采购订单（支持从发货需求自动预填 SKU 和数量的请购型，以及手动创建的备货型），关联供应商并补充单价，提交后进入最新采购状态机,
so that 采购流程可以快速启动，请购型自动继承发货需求信息并减少重复录入。

## Acceptance Criteria

1. 发货需求状态为 `ShippingDemandStatus.PENDING_PURCHASE_ORDER` 或 `ShippingDemandStatus.PURCHASING`，且存在 `purchaseRequiredQuantity > purchaseOrderedQuantity` 的明细时，后端提供可创建请购型采购订单的预填接口，前端在发货需求详情页启用“生成采购单”入口。（FR27, FR35）
2. PurchaseGroupPreview 采购分组预览面板使用 Drawer（约 520px），仅展示履行类型为“全部采购”或“部分采购”且可下单数量大于 0 的明细；每行默认数量 = `purchaseRequiredQuantity - purchaseOrderedQuantity`，用户本次下单数量不得超过该值。（UX-DR11, UX-DR27）
3. 用户确认采购分组后，系统按供应商分组批量创建一张或多张请购型采购订单；每张订单关联供应商、来源发货需求、合同条款、SKU/数量/单价，订单行必须记录 `shippingDemandId` 与 `shippingDemandItemId`。（FR27, FR35）
4. 后端在同一个 QueryRunner 事务内完成请购型采购订单创建、发货需求明细 `purchaseOrderedQuantity` 聚合回填、销售订单展示缓存必要回填、发货需求状态推进和 ActivityTimeline/操作审计记录；若发货需求原状态为 `pending_purchase_order`，创建成功后推进为 `purchasing`。（flow-quantity-data-lineage, flow-state-machine）
5. 请购型批量创建必须具备稳定幂等边界：重复点击或重复提交不得让同一请求批次重复生成采购订单，也不得重复累计 `purchaseOrderedQuantity`。
6. 采购订单状态必须遵守最新 flow 文档：创建后进入 `PurchaseOrderStatus.PENDING_CONFIRM`，后续确认动作按 `PENDING_CONFIRM -> SUPPLIER_CONFIRMING -> PENDING_RECEIPT` 推进；旧文档中的“创建即已确认/confirmed”口径不作为实现依据。（flow-state-machine 优先）
7. 手动进入采购订单创建表单时创建备货型采购订单；表单包含供应商、合同条款、备注和订单行项（SKU、数量、单价、金额），选择供应商后展示联系方式/账期等只读信息，合同条款默认选择已审批默认范本或第一条已审批范本。（FR35, UX-DR29）
8. 备货型采购订单创建不关联发货需求、不回填发货需求数量、不触发库存锁定；状态仍按最新采购订单状态机进入 `pending_confirm`。
9. 提供采购订单内部确认与供应商确认动作端点，禁止通过通用 PATCH 修改状态字段；动作成功后写入操作日志并返回最新订单。（flow-state-machine）
10. 后端采购订单实体包含：`id`、`poCode`、`supplierId`、`supplierName`、`shippingDemandId`（请购型关联，备货型为空）、`contractTermId`、`orderType`、`status`、`totalAmount`、`remark`、`createdAt`、`updatedAt`、`createdBy`、`updatedBy`；订单行包含 `id`、`purchaseOrderId`、`skuId`、`quantity`、`unitPrice`、`amount`、`shippingDemandId`、`shippingDemandItemId`、`receivedQuantity`、`createdAt`、`updatedAt`。
11. 前端 API 调用必须通过 `apps/web/src/api/` 层；页面使用 ProForm/ProTable/Drawer 等现有模式，不在组件内直接 axios，不重复定义 shared 枚举。
12. 后端单测覆盖：请购型预填筛选、数量上限校验、按供应商分组创建、状态推进到 `purchasing`、`purchaseOrderedQuantity` 聚合回填、幂等防重复、备货型创建、采购订单确认状态机动作；前端至少通过类型检查/构建验证 API 与页面类型正确。

- 发货需求确认分配含采购数量时只进入 `pending_purchase_order`；只有请购型采购订单创建成功后才推进到 `purchasing`。

- 请购型采购订单行必须关联 `shippingDemandId` 与 `shippingDemandItemId`，后续 Story 6.6 收货后自动锁定依赖该链路。
- 采购订单状态必须遵守 `flow-state-machine.md` 的最新状态链：`pending_confirm -> supplier_confirming -> pending_receipt -> partially_received/received`，不要沿用旧 `confirmed` 口径。
- 状态推进必须由领域动作服务在事务内执行；禁止前端直接提交目标状态，禁止通用 PATCH 修改状态字段。
- 禁止新增临时库存 API、重复枚举或绕过 `packages/shared` 的本地枚举。
