# Story 7.1: 物流单创建（从发货需求预填）

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 商务跟单,
I want 从发货需求详情页一键创建物流单，系统自动预填 SKU 和数量，我补充物流供应商、港口、装箱信息与报关备注，
so that 物流计划能够复用已锁定待发数据，减少重复录入，并为后续出库建立可靠的上游单据。

## Scope Adjustment

- Story 5.5 已落地“发货需求详情页跳转物流单创建页 + 物流单 MVP 持久化能力”，因此 Story 7.1 不应从零重建物流单模块，而应把现有 MVP 对齐到 Epic 7.1 正式口径。
- 本 Story 只负责“创建物流单”与“创建即确认”的闭环，不实现物流单列表/详情、物流跟踪后补、出库单创建、出库确认、物流费用。
- 物流计划数量只能来自 `lockedRemainingQuantity - activeLogisticsPlannedQuantity`；不得读取待采购数量，也不得扩大计划数量。
- 若现有 MVP 的数据建模与 Epic 7.1 AC 存在偏差，优先满足 AC 与三份 flow 文档；如需兼容已有数据，应在不破坏已合并 Story 5.5 的前提下平滑演进。

## Acceptance Criteria

1. 商务跟单在发货需求详情页点击“创建物流单”后，跳转到物流单创建表单；表单必须自动关联当前发货需求，且发货需求编号灰底只读不可编辑。
2. 货物明细表自动预填：SKU 编码、SKU 名称、可计划数量；数量口径必须是 `lockedRemainingQuantity - activeLogisticsPlannedQuantity`，且只读，不允许手工改来源需求或扩大计划数量。
3. 创建表单按以下分组呈现：
   - 基本信息：物流供应商（必填，EntitySearchSelect）、运输方式（必填，下拉：海运/陆运/空运/快递）、公司主体（必填，EntitySearchSelect）、起运港（必填，EntitySearchSelect）、目的港/目的地（必填，EntitySearchSelect）、运抵国（必填）。
   - 货物明细：来源字段、SKU、可计划数量只读；可编辑字段为箱数、每箱数量、长/宽/高（cm）、毛重（kg）。
   - 报关信息：是否出口报关、唛头。
   - 备注：备注文本。
4. 物流跟踪字段 `ETD` / `ETA` / `订舱号` / `船司·航司` / `船名航次` / `SO·提单号` / `实际离港日期` 必须存在于后端实体，但创建表单阶段不展示；物流单确认后由 Story 7.2 在详情页补填或更新。
5. 商务跟单提交物流单时使用 `Popconfirm` 二次确认；提交成功后物流单状态为 `LogisticsOrderStatus.CONFIRMED`，物流单编号自动生成，且库存数量不发生变化。
6. 后端物流单实体至少包含：`id`、`logistics_code/order_code`、`shipping_demand_id`、`logistics_provider_id`、`transport_mode`、`company_id`、`departure_port_id`、`destination_port_id`、`destination_country`、`etd`、`eta`、`booking_number`、`carrier`、`vessel_voyage`、`bl_so_number`、`actual_departure_date`、`is_customs_declaration`、`shipping_mark`、`status`、`remark`、审计字段。
7. 后端物流单货物行至少能表达：`logistics_order_id`、`shipping_demand_item_id`、`sku_id`、`quantity/planned_quantity`、`boxes`、`quantity_per_box`、`length_cm`、`width_cm`、`height_cm`、`gross_weight_kg`、审计字段；若继续沿用独立装箱表，必须保证 AC 级字段在接口和领域层语义上等价可追踪。
8. 并发场景下创建物流单必须继续沿用 `QueryRunner + SELECT ... FOR UPDATE + 指数退避最多 3 次`，并在重复计划或数量超限时给出明确业务错误。
