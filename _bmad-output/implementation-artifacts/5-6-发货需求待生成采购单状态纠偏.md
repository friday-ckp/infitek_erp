# Story 5.6: 发货需求待生成采购单状态纠偏

Status: done

<!-- Note: 本 Story 来自 2026-04-29 已批准 Sprint Change Proposal，必须在 Epic 6.3 开始前完成。 -->

## Story

As a 商务跟单,
I want 发货需求在确认分配后准确显示“待生成采购单”,
so that 我不会把尚未创建采购订单的需求误判为已经进入采购执行。

## Scope Adjustment

- 本 Story 是 Story 5.4/5.5 之后的 correct-course 补丁，修正“确认分配含采购数量后直接进入采购中”的业务语义错误。
- 新增发货需求中间状态 `pending_purchase_order`，含义是：库存分配已确认，存在采购缺口，但尚未创建任何有效请购型采购订单。
- 本 Story 必须更新 shared 枚举、后端确认分配状态推进、前端状态展示/筛选/FlowProgress/ActivityTimeline、文档口径和回归测试。
- 本 Story 不实现采购订单创建。采购订单创建、`purchaseOrderedQuantity` 回填、`pending_purchase_order -> purchasing` 状态推进由 Epic 6 Story 6.3 承接。
- 本 Story 不改变 Story 5.5 已落地的物流单 MVP 边界：物流单表单只能使用已锁定且未被 active 物流单计划的待发数量，不能使用待采购数量。

## Acceptance Criteria

1. `packages/shared` 中 `ShippingDemandStatus` 新增 `PENDING_PURCHASE_ORDER = 'pending_purchase_order'`，前端、后端、Jest shared mock 均通过 shared 枚举引用，不新增前后端独立枚举。
2. `POST /api/shipping-demands/:id/confirm-allocation` 在任一明细存在 `purchaseRequiredQuantity > 0` 时，将发货需求状态推进为 `pending_purchase_order`，不再直接推进为 `purchasing`。
3. `POST /api/shipping-demands/:id/confirm-allocation` 在全部明细都使用现有库存且库存锁定完成时，仍将发货需求推进为 `prepared`，并保持销售订单推进为 `prepared` 的现有行为。
4. 确认分配动作仍只允许 `pending_allocation` 状态调用；`pending_purchase_order`、`purchasing`、`prepared`、`partially_shipped`、`shipped`、`voided` 均不得再次确认分配或重复锁定库存。
5. 确认分配动作只写履行决策和锁定结果：`fulfillmentType`、`stockRequiredQuantity`、`purchaseRequiredQuantity`、`lockedRemainingQuantity`、allocation、库存流水和操作日志；不得创建采购订单，不得把采购缺口写成已下单数量，`purchaseOrderedQuantity` 维持既有聚合/默认值。
6. 发货需求列表页、详情页、编辑页、销售订单详情关联发货需求展示均支持“待生成采购单”状态标签、筛选项和状态样式；建议使用 warning/orange 语义色，但文案必须区别于“采购中”。
7. `ActivityTimeline` 的状态字段、操作日志详情和中文文案映射支持 `pending_purchase_order: 待生成采购单`，避免状态变更记录显示英文原值或未知状态。
8. 发货需求详情页 `FlowProgress` 在“分配库存”和“采购中”之间增加“待生成采购单”步骤；全库存路径允许从“分配库存”直接跳到“备货完成”。
9. 发货需求详情页采购 SmartButton 在 `pending_purchase_order` 状态展示下一步动作“生成采购单”。若 Epic 6 采购订单页面/API 尚未落地，按钮可保持禁用或占位，但文案必须清楚说明“采购订单功能将在 Epic 6 启用”，不能显示为“采购中”或暗示采购单已生成。
10. 发货需求详情页作废规则保持 Story 5.5 口径：只允许 `pending_allocation` 作废；`pending_purchase_order` 已离开待分配库存，不展示作废按钮。
11. 物流单创建后端和前端保持只读已锁定待发数量：`lockedRemainingQuantity - activeLogisticsPlannedQuantity`。`pending_purchase_order` / `purchasing` 中尚未收货锁定的待采购数量不得进入物流单可计划数量。
12. 单元测试覆盖：含采购数量确认分配后状态为 `pending_purchase_order`，全部库存确认分配后状态仍为 `prepared`，非待分配状态不能重复确认分配，物流单创建不能从 `pending_purchase_order` 使用待采购数量。
13. 文档口径更新完成：`prd.md`、`flow-state-machine.md`、`flow-cross-document-trigger.md`、`flow-quantity-data-lineage.md`、Story 5.4、Story 5.5、Epic 6.3、Epic 6.6、Epic 7.1、Epic 8.1 中与发货需求状态、采购单创建责任、物流数量来源冲突的描述均同步。
14. 验证通过：发货需求相关后端定向测试、物流单相关后端定向测试、API build、Web build 均通过；若全量测试存在既有无关失败，需在 Dev Agent Record 中记录。

### State Machine Target

发货需求目标状态口径：

| 状态 | 含义 | 进入动作 |
|---|---|---|
| `pending_allocation` | 已生成发货需求，尚未确认履行分配 | 销售订单详情手动生成发货需求 |
| `pending_purchase_order` | 已确认分配，存在采购缺口，但尚未创建有效请购型采购订单 | `confirm-allocation` 含任一采购数量 |
| `purchasing` | 已创建至少一张有效请购型采购订单，等待采购执行/收货/自动锁定 | Epic 6.3 采购订单创建成功 |
| `prepared` | 所有应发数量均已锁定或已发出，尚未全部出库完成 | 全库存确认分配，或 Epic 6.6 收货后自动锁定完成 |
| `partially_shipped` | 任一明细已出库但并非全部发完 | Epic 7 出库确认 |
| `shipped` | 所有明细全部出库完成 | Epic 7 出库确认 |
| `voided` | 作废终态 | 仅 `pending_allocation` 可作废 |

### Quantity Authority and Boundaries

- 发货需求明细是履约决策数量权威来源。
- `purchaseRequiredQuantity` 表示确认分配后仍需采购的数量。
- `purchaseOrderedQuantity` 表示未取消请购型采购订单累计下单数量；本 Story 不回填该字段，Epic 6.3 负责。
- Epic 6.3 可下单数量必须是 `purchaseRequiredQuantity - purchaseOrderedQuantity`。
- `lockedRemainingQuantity` 表示已锁定且未出库数量。
- 物流单可计划数量必须是 `lockedRemainingQuantity - activeLogisticsPlannedQuantity`，不得从 `purchaseRequiredQuantity` 或待采购缺口推导。
- 库存数量权威来源仍是 `inventory_summary`、`inventory_batch`、`shipping_demand_inventory_allocations`、`inventory_transactions`。

### Purchase Order Responsibility for Epic 6.3

本 Story 只暴露正确状态和入口，不实现采购订单。

Epic 6.3 必须承接以下规则：

- 仅允许从 `pending_purchase_order` 或 `purchasing` 的发货需求创建请购型采购订单。
- PurchaseGroupPreview 只展示 `purchaseRequiredQuantity > purchaseOrderedQuantity` 的发货需求明细。
- 默认可下单数量 = `purchaseRequiredQuantity - purchaseOrderedQuantity`。
- 创建成功后采购订单行必须关联 `shippingDemandId` 与 `shippingDemandItemId`。
- 创建成功后聚合回填 `purchaseOrderedQuantity`。
- 若发货需求原状态为 `pending_purchase_order`，且至少创建一张有效请购型采购订单，则推进为 `purchasing`。

### Architecture Compliance

- 所有枚举定义在 `packages/shared`，使用 `as const` 对象模式。
- 状态变更必须通过领域动作端点，不允许通用 PATCH 直接写 `status`。
- 确认分配是库存写动作，继续使用 TypeORM `QueryRunner` 事务、`InventoryService.lockStockInTransaction()` 和行锁。
- 不允许为了新状态新增临时库存 API、临时采购单占位表或前端硬编码状态事实。
- 操作日志/ActivityTimeline 只负责展示状态和动作，不作为状态权威来源。

### Implementation Boundaries

- 不实现采购订单模块、采购订单页面或采购订单创建接口。
- 不回填 `purchaseOrderedQuantity`，除非现有聚合器已经在其他动作中自然刷新；本 Story 不新增采购下单事实。
- 不允许 `confirm-allocation` 调用采购订单服务或创建采购订单草稿。
- 不放宽物流单创建状态到 `pending_purchase_order` / `purchasing`；MVP 继续只允许 `prepared` / `partially_shipped`。
- 不允许物流单预填或提交待采购数量。
- 不恢复或改写 Story 5.5 已完成的物流 MVP 入口和作废闭环。
- 不删除历史文档；更新文档时保留变更记录或直接修正文案。
