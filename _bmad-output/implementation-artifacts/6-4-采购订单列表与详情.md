# Story 6.4: 采购订单列表与详情

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 授权用户,
I want 查看采购订单列表（按状态和供应商筛选）和订单详情,
so that 我可以跟踪采购进度和查看订单明细。

## Acceptance Criteria

1. 授权用户进入 `/purchase-orders` 后，展示正式采购订单列表页（ProTable），替换 Story 6.3 的占位页；列表字段从用户确认的“主数据”字段中选取核心跟踪字段，至少包含：采购订单号（蓝色链接）、供应商编码、供应商名称、采购状态、申请类型、需求类型、销售订单号、发货需求编号、采购日期、PO交期、到货日期、采购总数量、采购金额合计、已入库总数、入库状态、创建时间。
2. 列表支持快捷关键词搜索、采购状态筛选、供应商筛选、申请类型筛选、需求类型筛选、入库状态筛选与分页；筛选参数通过 `apps/web/src/api/purchase-orders.api.ts` 调用后端 `GET /purchase-orders`，组件内不得直接 axios。
3. 列表页遵守 UX-DR04：页面标题与主操作按钮、快捷搜索 + 高级筛选、已选条件 Tag、固定表头、48px 行高、固定右侧操作列、底部分页器与总数。
4. 点击订单编号或行操作“查看”进入 `/purchase-orders/:id` 详情页；详情页按 UX-DR05 展示摘要卡片、锚点导航、信息分区和状态 pill。
5. 采购订单详情页展示用户确认的“主数据”全部字段，并展示用户确认的“采购产品明细”全部字段；现有上游链路暂时无法自动获得的字段必须以可空字段落库并展示为空值，不得在前端伪造。
6. 状态展示必须使用 `packages/shared` 的 `PurchaseOrderStatus` 与采购订单相关 shared 枚举，中文文案只在 UI 映射层处理；不得新增本地重复枚举。用户确认的“已开票”状态需要补入 shared 枚举，但不得改变既有 `pending_confirm -> supplier_confirming -> pending_receipt -> partially_received/received` 交易状态机动作。
7. 详情页可展示 Story 6.3 已有的内部确认与供应商确认动作入口：仅当状态分别为 `pending_confirm` / `supplier_confirming` 时显示，并通过既有动作 API 调用，不允许通用 PATCH 修改状态。
8. 后端列表与详情接口保持只读查询语义；本 Story 不新增库存写入、不改采购数量权威来源、不创建收货入库单、不推进收货状态。
9. 后端测试覆盖采购订单列表查询条件与详情读取；前端至少通过类型检查/构建验证，并补充 API 层类型和页面路由验证。

### UI / UX Guidance

- 列表复用 `SalesOrdersListPage` / `ShippingDemandsListPage` 的 `SearchForm` + `ProTable` 模式。
- 详情复用销售订单/发货需求详情的 `master-summary-card`、`AnchorNav`、`SectionCard`、`MetaItem`、`SummaryMetaItem`、`ActivityTimeline`。
- 状态色彩建议：
  - `pending_confirm`: 默认/灰色（待确认）
  - `supplier_confirming`: warning/橙色（待供应商确认）
  - `pending_receipt`: processing/蓝色（待收货）
  - `partially_received`: warning/橙色（部分收货）
  - `received`: success/绿色（全部收货）
  - `cancelled`: error/红色（已取消）
- 订单类型建议使用 Tag：请购型/备货型。
- 金额使用两位小数和千分位；空值统一展示 `—`。
