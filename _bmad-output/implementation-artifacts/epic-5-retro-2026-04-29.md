## 交付概览

Epic 5 从单据录入推进到第一段交易履约链路，实际交付范围包括：

- 销售订单创建、提交审核、审核通过、驳回、作废和详情操作记录。
- 库存双层结构前置：`inventory_summary`、`inventory_batch`、可用库存查询和期初库存录入。
- 审核通过销售订单手动生成发货需求，并推进销售订单至备货中。
- 发货需求列表、详情、销售订单镜像字段、附件签名 URL、KPI 和 FlowProgress。
- 发货需求确认分配闭环：履行类型编辑、确认分配动作端点、FIFO 锁定、批次级 allocation、库存流水和操作日志。
- 发货需求完整作废闭环：释放 allocation、写 unlock 库存流水、回退销售订单、保留旧作废单据并支持重新生成。
- 物流单 MVP 创建入口和最小持久化能力，为 Epic 7 链路提前打底。

## 主要返工点

### 1. 作废被低估为状态修改

作废不是把发货需求状态改成 `voided`。在本组需求里，作废是完整反向交易动作：

- 只允许 `pending_allocation` 作废。
- 已作废再次作废必须幂等，不重复释放库存。
- 若存在 active allocation，必须通过 `InventoryService.unlockStockInTransaction()` 解锁。
- allocation 状态要从 active 释放为 released，数量字段要回填。
- 必须写 `inventory_transactions` unlock 流水。
- 发货需求要记录 `voidedAt` / `voidedBy`。
- 销售订单状态要回退为 `approved`。
- 销售订单详情要重新允许生成新的发货需求。
- 旧发货需求不能删除，必须保留审计。

返工原因：早期 Story 更像页面/按钮补齐，没有在 Story 创建阶段列出“反向动作”的完整数据影响面。

### 2. 编辑规则缺少状态 × 字段矩阵

销售订单和发货需求都不是普通 CRUD 编辑：

- 销售订单已生成过发货需求后，不允许修改产品 SKU 和数量。
- 销售订单在履约链路中只能开放有限业务字段修改。
- 发货需求编辑只能修改业务资料，不允许修改状态、产品数量、履约数量和受保护字段。
- 已作废销售订单/发货需求不允许编辑。
- 履行类型编辑不应走普通 update，而应纳入确认分配动作。

返工原因：编辑入口在 UI 上容易被视作“详情页常规按钮”，但交易单据需要先定义可编辑字段矩阵，再实现前后端更新逻辑。

### 3. 操作日志验收粒度不够

“详情页展示 ActivityTimeline”不等于操作日志完成。Epic 5 暴露出更细的日志要求：

- 销售订单状态推进要写销售订单操作记录。
- 生成发货需求要能在销售订单和发货需求上下文里被追溯。
- 确认分配要记录履行类型、状态变化和库存锁定摘要。
- 作废要记录发货需求作废和销售订单回退。
- ActivityTimeline 需要补齐 resource、field label、枚举中文化和关联单据展示。

返工原因：Story 早期把操作记录当成“展示组件”，没有把它当成每个领域动作的验收项。

### 4. Story 5.4 范围误读

5.4 一度被实现为“发货需求详情字段补齐 + 只读枢纽展示”，但 Epic 5.4 原始验收标准要求“含完整库存锁定”：

- 履行类型选择。
- 使用库存数量和仓库选择。
- FIFO 批次预览。
- 确认分配按钮。
- 后端确认分配动作端点。
- FIFO 锁定库存。
- 写 allocation 和库存流水。
- 推进发货需求和销售订单状态。
- 操作记录刷新。

返工原因：Story 实现边界主动收窄，但没有及时对照 Epic AC 和 flow 文档确认“库存锁定”是否可后移。5.4 纠偏后，5.5 作废才具备可消费的 allocation 和库存流水基础。

## 根因总结

Epic 5 的真实复杂度不在页面字段，而在跨单据交易一致性：

- 销售订单、发货需求、库存、allocation、库存流水、物流单之间存在状态联动。
- 数量事实有权威来源，不能由多个单据各自随意写。
- 正向动作必须同时设计反向动作。
- 操作日志是业务可追溯能力，不是页面附属展示。

因此，按“列表页/详情页/按钮”拆 Story 容易漏掉领域动作的完整影响面。Epic 5 后半段通过 flow 文档、纠偏提案和 code review 把方向拉回了“领域动作驱动”。

## 技术规范沉淀

### 后端

- 状态变化必须使用专用动作端点，禁止通用 PATCH 直接改状态。
- 交易动作必须在 Service 层事务内完成，跨库存动作使用 QueryRunner 和既有 InventoryService。
- 数量权威来源遵守 flow 文档：库存汇总、库存批次、发货需求分配表、库存流水。
- 作废/取消/回滚类动作必须幂等，并使用稳定 `sourceActionKey` 避免重复流水。
- 每个领域动作必须同步写 OperationLog，状态联动影响上游单据时也要写上游单据日志。

### 前端

- 交易单据编辑按钮必须受状态和字段矩阵约束。
- 不可逆动作使用明确二次确认，并说明状态、库存和上游单据影响。
- 成功动作必须刷新详情、列表、操作记录和相关上游详情缓存。
- ActivityTimeline 是详情页标准组成，不新增分散的审计信息区块。
- 关联下游入口必须避免跳转不存在路由；未实现时保持禁用占位或最小可用路径。
