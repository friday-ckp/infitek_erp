# Story 9.3: 前端登录页与钉钉回调页

Status: done

## Story

As a 企业员工,
I want 在登录页选择账号密码或钉钉扫码登录，并在授权返回后自动完成登录,
so that 我的登录体验更贴近企业日常工作流，同时保留原有密码登录兜底。

## Acceptance Criteria

1. **登录页双入口**
   - 用户未登录进入系统时，登录页同时展示“账号密码登录”和“钉钉扫码登录”入口
   - 现有用户名/密码登录表单继续可用，不因 SSO 接入而降级

2. **钉钉登录入口接通**
   - 用户点击“钉钉扫码登录”后，浏览器跳转到后端提供的钉钉登录入口 `/api/auth/dingtalk/login`
   - 不再显示“该功能暂未开发”的占位提示

3. **前端回调自动完成登录**
   - 钉钉授权成功后跳转到 `/login/dingtalk/callback?ticket=...`
   - 回调页自动调用 ticket 交换接口 `POST /api/auth/dingtalk/exchange`
   - 成功后把 JWT 与最小用户信息写入现有认证 store
   - 成功后跳转到系统主页 `/` 或原目标受保护页面

4. **失败场景可操作**
   - 回调页缺少 ticket、ticket 已过期、ticket 已使用、ticket 无效或后端返回未绑定错误时，页面展示清晰错误提示
   - 未绑定场景明确提示联系管理员完成钉钉绑定
   - 用户可返回登录页并改用密码登录或重新发起钉钉登录

5. **登录态与受保护路由一致**
   - 钉钉登录成功后，刷新页面或访问受保护路由时，后续请求继续自动携带 JWT
   - 受保护页面行为与密码登录完全一致，不新增第二套登录态逻辑

6. **符合现有 UX 与安全边界**
   - 登录页与回调页在 loading、失败、重试状态下提供符合现有视觉语言和可访问性语义的反馈
   - 前端不得暴露原始异常对象、内部堆栈或第三方原始返回
   - JWT 不得通过 URL 暴露

## Dev Notes

### Story Boundary

- 本 Story 只实现前端登录页双入口、钉钉回调页、现有 auth store 接入、受保护路由回跳和前端错误态。
- 本 Story 不修改后端钉钉 OAuth 协议、不新增绑定/解绑接口、不实现管理员绑定 UI；这些属于 Story 9.4。
- 本 Story 不新增 refresh token、SSO 会话中心、二维码嵌入式钉钉 SDK 或第二套登录态模型。
- 除非实现前端可用性被现有后端契约阻塞，否则不要改动 Story 9.2 已完成的 `/api/auth/dingtalk/login`、`/api/auth/dingtalk/callback`、`/api/auth/dingtalk/exchange` 契约。

### Story Foundation And Cross-Story Context

- Epic 9 的目标是在保留账号密码登录的前提下，引入“钉钉只负责外部认证，本地系统继续签发 JWT”的企业 SSO 模式。
- Story 9.2 已完成后端主链路，当前前端必须直接消费三个后端入口：
  - `GET /api/auth/dingtalk/login`
  - `GET /api/auth/dingtalk/callback`
  - `POST /api/auth/dingtalk/exchange`
- Story 9.4 会在用户管理中补上绑定/解绑能力，因此 9.3 对“未绑定”场景只能提示联系管理员，不能偷偷实现自动开户或临时绑定。
- Story 9.5 会以本 Story 的页面行为做联调和上线验证，所以 redirect、错误码映射、密码登录兼容性必须稳定。

### Current Code Context

- `apps/web/src/pages/login/index.tsx` 目前已有完整的用户名/密码登录表单和双栏视觉布局，但 SSO 按钮仍只是 `antdStatic.message?.info('该功能暂未开发')` 占位。
- `apps/web/src/store/auth.store.ts` 当前只保存：
  - `token`
  - `user: { username, name }`
  - `login(token, user)`
  - `logout()`
  钉钉登录成功后必须复用这一路径，不要新建第二个 store。
- `apps/web/src/api/request.ts` 已完成：
  - `baseURL: '/api'`
  - 自动注入 `Authorization: Bearer <token>`
  - 401 时默认跳转 `/login`
  - 成功响应只返回 `response.data.data`
  - 普通失败时会触发全局 `message.error`
  因此新增 auth API 只需要返回业务 `data` 类型，不要手工再包一层响应壳；但 callback 页需要避免全局错误提示与页面级错误态重复。
- `apps/web/src/components/layout/AppLayout.tsx` 当前 token 无效时直接 `<Navigate to="/login" replace />`，不会保留用户原始目标页；这与本 Story AC 3 的“原目标页面”要求冲突，必须修正。
- `apps/web/src/App.tsx` 当前只有 `/login` 是公开路由，尚未注册 `/login/dingtalk/callback`。
- `apps/web/src/App.tsx` 的 QueryClient 默认 `retry: 1`。这对普通 GET 可以接受，但对一次性 `loginTicket` 的 exchange 是陷阱：自动重试会把真实失败放大成 `DINGTALK_TICKET_USED`。
- `apps/web/src/api/request.ts` 当前 401 重定向和 `AppLayout` 的 token 校验都不会保留 `search/hash`，如果只保存 `pathname`，将来带查询参数的详情页回跳会丢上下文。

### Frontend Guardrails

- 继续使用当前真实 API 前缀 `/api`，不要回退到旧文档中的 `/api/v1` 口径。[Source: `apps/web/src/api/request.ts`, `apps/api/src/main.ts`]
- 钉钉登录按钮应直接把浏览器送到后端登录入口，不要在前端自己拼钉钉 OAuth URL；`state` 由后端负责生成与消费。[Source: `_bmad-output/implementation-artifacts/9-2-后端钉钉OAuth回调与JWT签发.md`]
- `ticket` 是一次性的。前端不得：
  - 重试 exchange
  - 并发触发多次 exchange
  - 把 `ticket` 长期写入 localStorage/sessionStorage
- 登录成功后的前端持久化仍然只有：
  - `localStorage.token`
  - `localStorage.user`
  不要新加 “dingtalkToken”“ticketStore”等旁路持久化。
- redirect 目标必须做白名单化校验，只允许站内相对路径，否则存在开放重定向风险。
- redirect 保留必须包含 `pathname + search + hash`，否则用户从筛选后的列表页或带查询参数的创建页跳登录后会丢失上下文。
- 页面错误展示只允许使用稳定业务码映射后的友好文案，不允许把原始 `error.response`、堆栈或第三方字段直接渲染到 UI。

### Architecture Compliance

- 认证策略仍是单一 Access Token（8h、无 Refresh Token）；密码登录和钉钉登录最终都写入同一种系统 JWT。[Source: `_bmad-output/planning-artifacts/architecture.md#认证与安全`]
- 后端 OAuth 回调成功后不直接在 URL 上返回 JWT，而是先返回一次性 `loginTicket`；前端必须按此顺序消费，不能尝试把 token 放进 URL。[Source: `_bmad-output/planning-artifacts/architecture.md#认证与安全`]
- 受保护页面依赖全局 `Authorization` 注入和统一 401 处理，Story 9.3 必须继续复用该链路，而不是为钉钉登录写一套“只对回调页生效”的特例鉴权。[Source: `apps/web/src/api/request.ts`]
- 系统当前的服务器状态管理标准是 TanStack Query，不能退回为纯手写请求状态机。[Source: `_bmad-output/planning-artifacts/architecture.md#前端架构`, `_bmad-output/project-context.md#Frontend Framework Rules`]

### UX And Accessibility Notes

- 当前登录页已经形成独立的品牌化双栏视觉风格。Story 9.3 应在这个基础上增量扩展，不要把页面整体重做成普通后台表单页。
- 回调页可以复用登录页的视觉壳或相近的版式，但状态反馈必须明确区分：
  - 正在完成登录
  - 登录成功即将跳转
  - 登录失败，可返回或重试
- 失败态至少应提供：
  - 页面级错误标题
  - 解释文案
  - 返回登录页按钮
  - 对未绑定场景的联系管理员提示
- 为按钮、加载区域、错误提示补齐可访问性语义，例如：
  - 按钮 `aria-label`
  - loading 状态可感知文案
  - 错误提示继续使用 antd `Alert` 或等价语义化组件

### Recommended Implementation Shape

- 建议新增 `apps/web/src/api/auth.api.ts`，例如：
  - `exchangeDingtalkTicket(ticket: string)`
  - 若需要，也可提供 `getDingtalkLoginPath(): string`，但不需要额外请求后端拿 URL
- 建议新增一个极小的 redirect 工具模块，例如 `apps/web/src/utils/auth-redirect.ts`，统一处理：
  - `buildLoginRedirectUrl(currentPath)`
  - `resolvePostLoginRedirect(searchParams)`
  - `isSafeRedirectTarget(value)`
- `buildLoginRedirectUrl(...)` 推荐直接基于 `location.pathname + location.search + location.hash` 生成，避免各调用点自行拼接不一致。
- 建议在回调页用 TanStack Query 实现“自动 exchange + 页面级状态”，但必须关闭重试：
  - `retry: false`
  - `refetchOnWindowFocus: false`
  - `refetchOnReconnect: false`
- 若改 `request.ts`，优先选择最小改法：
  - 对 `/auth/login` 和 `/auth/dingtalk/exchange` 豁免自动 401 跳转
  - 或增加可选请求配置以 suppress 401 redirect / toast
  不要顺手重构所有 API 错误处理。

### Error Contract Guidance

- 后端 callback 失败时，前端收到的是 `/login/dingtalk/callback?error=...`，不是 JSON 响应；页面必须先处理 query 中的 `error`。
- 后端 exchange 失败时，前端拿到的是当前统一错误结构中的 `message` 和 `code`；UI 应优先按 `code` 映射文案。
- 推荐前端错误文案口径：
  - `DINGTALK_ACCOUNT_UNBOUND` → `当前钉钉账号尚未绑定 ERP 用户，请联系管理员完成绑定。`
  - `DINGTALK_ACCOUNT_DISABLED` → `当前账号已停用，请联系管理员处理。`
  - `DINGTALK_TICKET_EXPIRED` → `本次钉钉登录已过期，请重新扫码登录。`
  - `DINGTALK_TICKET_USED` → `本次钉钉登录已失效，请重新扫码登录。`
  - `DINGTALK_TICKET_INVALID` / `DINGTALK_STATE_INVALID` / `DINGTALK_OAUTH_FAILED` → `钉钉登录失败，请重试或改用账号密码登录。`

### Technical Research Summary

- 推荐登录时序已经确定为：
  - 登录页点击钉钉按钮
  - 浏览器进入钉钉授权

  - `/login/dingtalk/callback?error=DINGTALK_ACCOUNT_UNBOUND` 显示联系管理员文案
  - 过期/已使用 ticket 不会被全局 401 强制吃掉，而是保留在回调页展示页面级错误
- 推荐验证命令：
  - `pnpm --filter web build`
  - `pnpm --filter web test:e2e`
- 如果只做最小验证，也要明确区分：
  - 本 Story 新增的前端回归
  - 仓库原有无关 E2E 问题（如有）

### Project Structure Notes

- 登录回调页是认证专项公共页，放在 `apps/web/src/pages/login/` 下最自然；不要为这一页新建一整套 `modules/auth/` 目录树。
