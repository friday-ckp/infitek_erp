# Story 9.6: 钉钉组织用户同步与用户池建模

Status: done

## Story

As a 系统管理员,
I want 手动同步钉钉组织用户到 ERP 并形成待处理用户池,
so that 我不需要手工查找和录入 `dingtalkUnionId`，就能为扫码登录准备稳定的绑定基础。

## Acceptance Criteria

1. **管理员可手动同步钉钉组织用户**
   - 系统提供管理员可调用的同步入口
   - 同步时后端调用钉钉组织用户接口拉取当前组织用户清单
   - 同步结果写入独立的钉钉用户池，而不是直接创建 ERP 用户

2. **钉钉用户池支持幂等 upsert**
   - 再次同步同一钉钉身份时，不重复插入
   - 系统按稳定身份键更新展示信息与最近同步时间
   - 同步结果可统计新增、更新、已存在等摘要

3. **用户池具备最小必要字段与状态**
   - 每条记录至少保存：
     - 稳定身份键（优先 `unionId`）
     - 昵称
     - 手机号、邮箱、工号（若钉钉接口可提供）
     - 部门或组织展示信息（若钉钉接口可提供）
     - 最近同步时间
   - 每条记录具备状态字段，至少支持：
     - `UNBOUND`
     - `CANDIDATE`
     - `CONFLICT`
     - `BOUND`

4. **保持“只同步、不自动开户、不自动绑定”**
   - 本 Story 只建立钉钉用户池，不直接创建本地 ERP 用户
   - 本 Story 不直接把同步结果回写到 `users.dingtalk_*`
   - 本 Story 不自动生成正式绑定关系

5. **仅管理员可执行同步**
   - 当前仓库现实下，继续使用 `currentUser.username === 'admin'` 作为临时管理员判定
   - 非 `admin` 用户调用同步接口时返回 `403`
   - 前端如暴露同步入口，非 `admin` 用户应隐藏或禁用

6. **错误处理与安全边界正确**
   - 钉钉接口失败、返回异常或配置缺失时，系统返回统一错误响应
   - 不泄露钉钉 access token、client secret 或原始异常堆栈
   - 不影响现有扫码登录、密码登录和手工绑定主链路

## Dev Notes

### Story Boundary

- 本 Story 只负责“同步钉钉组织用户并建立独立用户池”。
- 本 Story 不实现正式绑定确认流程，那是 Story 9.7 / 9.8 的职责。
- 本 Story 不自动开户、不自动绑定、不改造现有扫码登录回调链路。
- 本 Story 不回退 Story 9.4 已交付的手工绑定能力；手工绑定仍然是兜底方案。

### Story Foundation And Cross-Story Context

- Story 9.1 已把 `users` 表扩展为正式绑定结果承载体，并明确“管理员预绑定，不自动开户”的原则。
- Story 9.2 已完成扫码登录回调、一次性 `loginTicket` 和 JWT 签发，运行时只依赖 `users.dingtalkUnionId` 做正式绑定查找。
- Story 9.4 已提供手工绑定/解绑能力，但当前仍要求管理员知道 `dingtalkUnionId`，这正是 9.6 要解决的运营门槛。
- Story 9.5 正在做上线验收，但真实组织环境仍缺少“同步组织用户 -> 生成候选 -> 管理员确认”的运营链路，因此需要新增 9.6-9.9 补强 Epic 9。

### Previous Story Intelligence

- Story 9.4 已明确临时管理员规则必须是 `currentUser.username === 'admin'`，9.6 应继续沿用，避免虚构 RBAC。
- Story 9.4 已明确不要把钉钉字段塞进通用 `PATCH /users/:id`；9.6 也不应把候选池逻辑揉进 `users` 通用 CRUD。
- 最近相关提交顺序为：
  - `feat(story-9.1): 钉钉应用配置与用户模型扩展`
  - `feat(story-9.2): 后端钉钉OAuth回调与JWT签发`
  - `feat(story-9.3): 前端登录页与钉钉回调页`
  - `feat(story-9.4): 用户管理钉钉绑定能力`
- 这说明 Epic 9 的既有主线是“模型 -> 认证链路 -> 前端入口 -> 手工绑定”，9.6 应继续沿着这条主线补“后台同步池”，而不是重写 auth 流程。

### Current Code Context

- 当前扫码登录钉钉客户端位于 `apps/api/src/modules/auth/dingtalk-auth.client.ts`，只覆盖 OAuth 授权与当前登录用户资料获取。
- 当前正式绑定结果位于 `apps/api/src/modules/users/entities/user.entity.ts` 的 `dingtalk_*` 字段。
- 当前手工绑定接口位于：
  - `apps/api/src/modules/users/users.controller.ts`
  - `apps/api/src/modules/users/users.service.ts`
- 当前前端绑定展示位于：
  - `apps/web/src/pages/settings/users/index.tsx`
  - `apps/web/src/pages/settings/users/detail.tsx`
- 当前没有独立的钉钉用户池实体、repository、service 或 controller，这正是本 Story 的主要新增点。

### Architecture Compliance

- 当前真实 API 前缀是 `/api`，不是旧文档中的 `/api/v1`；新接口必须遵循 `/api/...` 事实。[Source: `_bmad-output/project-context.md#Critical Don't-Miss Rules`]
- 认证体系仍是统一 Access Token（8 小时、无 Refresh Token）；9.6 不能引入旁路登录态。[Source: `_bmad-output/planning-artifacts/architecture.md#认证与安全`]
- 扫码登录主链路继续只依赖 `users.dingtalk_union_id` 查找正式绑定关系；9.6 建立的用户池只作为后台运营数据，不直接参与登录放行。[Source: `_bmad-output/planning-artifacts/architecture.md#认证与安全`]
- 钉钉用户池与正式绑定关系需要分层，避免未确认候选污染正式登录主链路。[Source: `_bmad-output/planning-artifacts/architecture.md#认证与安全`]

### Data Model Guidance

- 推荐新增独立表，如 `dingtalk_org_users`，至少包含：
  - `unionId` 或等价稳定身份键
  - `userId`
  - `openId`
  - `nick`
  - `mobile`
  - `email`
  - `jobNumber`
  - `departmentNames` 或等价组织展示字段
  - `status`
  - `suggestedUserId`
  - `matchReason`
  - `lastSyncedAt`
  - `lastProcessedAt`
  - `processedBy`
- 若钉钉接口在当前阶段无法稳定提供部分字段，允许字段 nullable，但表结构要为后续 9.7/9.8 预留。
- 状态枚举建议统一使用英文值：`UNBOUND`、`CANDIDATE`、`CONFLICT`、`BOUND`。

### Recommended Implementation Shape

- 推荐新增模块：
  - `apps/api/src/modules/dingtalk-org-users/`
- 该模块可包含：
  - `dingtalk-org-users.controller.ts`
  - `dingtalk-org-users.service.ts`
  - `dingtalk-org-users.repository.ts`
  - `entities/dingtalk-org-user.entity.ts`
  - `dto/query-dingtalk-org-users.dto.ts`
  - `tests/`
- 若需要访问钉钉组织接口，推荐新增专用 client，例如：
  - `apps/api/src/modules/dingtalk-org-users/dingtalk-org-users.client.ts`
- 不建议把组织同步逻辑直接堆到 `AuthService` 或 `UsersService` 中。

### Testing Guidance

- 后端测试重点验证：
  - 同步接口权限
  - upsert 幂等
  - 唯一索引与重复同步行为
  - 外部接口失败时的统一错误结构
- 建议命令：
  - `pnpm --filter api test -- --runInBand auth`
  - `pnpm --filter api test -- --runInBand users`
  - `pnpm --filter api build`
  - `git diff --check`
